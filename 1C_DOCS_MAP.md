# Карта документов 1С

Этот файл нужен, чтобы быстро понимать, какой документ за что отвечает, и не читать все подряд.

## 1. Минимальный рабочий набор

Для обычной задачи по 1С достаточно такого набора:
1. `CODEX_MISTAKES_1C.md`
2. `PATTERNS_1C.md`
3. локальный эталон в проекте
4. `RULES_1C_QUICK.md`

Если задача идет в строгом режиме проекта, добавляется:
5. `CODEX_STRICT_1C.md`

Если задача сложная, нетиповая или затрагивает архитектуру:
6. `RULES_1C.md`

## 2. Роли документов

### 2.1 Основные

- `RULES_1C_QUICK.md`
  - обязательное ядро;
  - короткие правила ежедневной работы;
  - основной стандарт по умолчанию.

- `RULES_1C.md`
  - полный стандарт;
  - нужен для сложных, нетиповых и промышленных сценариев;
  - содержит расширенные правила, профили, приложения и чек-листы.

- `CODEX_STRICT_1C.md`
  - жесткий локальный протокол;
  - усиливает процесс работы именно в этом проекте;
  - задает порядок чтения, приоритеты и мини-чек перед правками.

### 2.2 Память проекта

- `CODEX_MISTAKES_1C.md`
  - журнал реальных ошибок;
  - источник антипримеров;
  - нужен первым, чтобы не повторять уже пойманные промахи.

- `PATTERNS_1C.md`
  - журнал успешных решений;
  - источник локальных эталонов;
  - нужен после ошибок и перед новой реализацией.

### 2.3 Служебные

- `PROMPT_STRICT_1C.txt`
  - короткая готовая фраза для включения строгого режима в запросе.

- `CODEX.md`
  - короткий корневой протокол проекта по 1С;
  - не дублирует полный стандарт;
  - направляет к нужным документам.

## 3. Какой документ открывать по ситуации

### 3.1 Если задача обычная

Открывать:
1. `CODEX_MISTAKES_1C.md`
2. `PATTERNS_1C.md`
3. `RULES_1C_QUICK.md`

### 3.2 Если задача по XML, формам, метаданным, загрузке конфигурации

Открывать:
1. `CODEX_STRICT_1C.md`
2. `CODEX_MISTAKES_1C.md`
3. `PATTERNS_1C.md`
4. `RULES_1C_QUICK.md`
5. `RULES_1C.md`

### 3.3 Если задача учебная

Основной упор:
- `RULES_1C_QUICK.md`
- `PART B` в `RULES_1C.md`

### 3.4 Если задача промышленная или сложная

Основной упор:
- `RULES_1C.md`
- `PART C`
- `PART D`
- приложения и чек-листы

## 4. Что теперь можно не делать

Чтобы не терять время:
- не нужно читать все документы подряд перед каждой задачей;
- не нужно держать дублирующиеся правила в нескольких файлах;
- не нужно использовать `CODEX.md` как полный стандарт по 1С;
- не нужно копировать длинные правила в каждый новый запрос.

## 5. Источники истины по приоритету

При конфликте ориентируйся так:
1. реальные файлы проекта;
2. `CODEX_MISTAKES_1C.md`;
3. `PATTERNS_1C.md`;
4. `CODEX_STRICT_1C.md`;
5. `RULES_1C_QUICK.md`;
6. `RULES_1C.md`;
7. внешние источники.

## 6. Короткая формула для пользователя

Если нужен строгий режим, достаточно писать:

`Работай строго по CODEX_STRICT_1C.md: сначала CODEX_MISTAKES_1C.md, потом PATTERNS_1C.md, потом локальный эталон, потом одна реализация без архитектурных прыжков; если эталона нет, честно остановись и скажи, что нужен образец.`

## 0.4 Протокол взаимодействия AI

1. **Анализ задачи:** Если требования размыты, задай уточняющие вопросы ПЕРЕД генерацией кода.
2. **Self-Check — БЛОКИРУЮЩЕЕ ПРАВИЛО:** Перед выдачей результата AI **обязан** последовательно пройти все применимые чек-листы из Part D. **Каждый пункт каждого чек-листа должен быть отмечен как выполненный (☑).** Если хотя бы один пункт нарушен — код **НЕДОПУСТИМ** к выдаче. AI **НЕ ИМЕЕТ ПРАВА** выдать код с любым нарушением чек-листа, даже «временным», «для примера» или «с пометкой». Нарушение = переписать код до полного соответствия.
3. **Контекст:** Используй суффиксы модулей БСП и правила именования строго.
4. **Безопасность:** Никогда не генерируй пароли в открытом виде. Используй `ЗащищённоеХранилище`.
5. **Формат:** Код должен быть готов к вставке (copy-paste). Избегай комментариев вида "тут ваш код".

## 0) Мандат документа

### 0.1 Как использовать этот документ

- **AI-агент** обязан следовать правилам из разделов Part A–Part D при генерации любого кода, запроса, конфигурации или отчёта.
- **Каждое правило** имеет уровень строгости: `MUST` (обязательно), `SHOULD` (рекомендовано), `MUST NOT` (запрещено).
- **Перед выдачей результата** AI проходит Self-Check (раздел Part D). **Код с любым нарушением чек-листа НЕ ДОЛЖЕН быть выдан пользователю — это блокирующее ограничение.**
- **Если правило противоречит задаче** — AI обязан явно указать противоречие и предложить альтернативу.

### 0.2 Уровни строгости

| Уровень | Значение | Последствие нарушения |
|---------|----------|----------------------|
| `MUST` | Обязательно | Код считается невалидным |
| `SHOULD` | Рекомендовано | Требует обоснования отступления |
| `MUST NOT` | Запрещено | Критическая ошибка, немедленный отказ |

# PART A — ДАННЫЕ И ОТЧЁТЫ

## 1. Запросы: фундаментальные правила

### 1.1 Выборка полей

- **[A1-R1] MUST** Выбирать только явно указанные поля. `ВЫБРАТЬ *` запрещён.
- **[A1-R2] MUST** Использовать алиасы (`КАК`) для всех полей.

```bsl
// ✅ ПРАВИЛЬНО
"ВЫБРАТЬ
|   Т.Ссылка     КАК Ссылка,
|   Т.Наименование КАК Наименование,
|   СУММА(Т.Сумма) КАК Сумма
|ИЗ Документ.Реализация КАК Т"

// ❌ НЕПРАВИЛЬНО
"ВЫБРАТЬ * ИЗ Документ.Реализация"
```

### 1.2 Параметры

- **[A1-R3] MUST** Все значения передавать через параметры (`&Имя`), никогда — конкатенацией.
- **[A1-R4] MUST** Устанавливать параметры через `Запрос.УстановитьПараметр()`.

```bsl
// ✅ ПРАВИЛЬНО
Запрос.УстановитьПараметр("ДатаНач", ДатаНач);

// ❌ НЕПРАВИЛЬНО
Текст = "ГДЕ Дата > '" + Формат(ДатаНач, "ДФ=yyyyMMdd") + "'";
```

### 1.3 Отборы

- **[A1-R5] MUST** Отборы размещать как можно ближе к источнику (в `ГДЕ` или в условиях соединений).
- **[A1-R6] MUST** Использовать индексированные поля в отборах.
- **[A1-R7] MUST NOT** Применять функции к полям в `ГДЕ` без крайней необходимости (`ГДЕ Год(Дата) = 2024` → `ГДЕ Дата МЕЖДУ '20240101' И '20241231'`).

### 1.4 Соединения

- **[A1-R8] SHOULD** Минимизировать количество соединяемых таблиц.
- **[A1-R9] MUST** Использовать `ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ` только когда альтернативы нет (ИТС #std435).
- **[A1-R10] SHOULD** Предпочитать `ВНУТРЕННЕЕ СОЕДИНЕНИЕ` когда правая часть гарантированно существует.
- **[A1-R10.1] SHOULD** Использовать `ЛЕВОЕ СОЕДИНЕНИЕ`, когда нужно сохранить полный набор записей левого источника.
- **[A1-R10.2] MUST** Учитывать, что порядок и вложенность соединений при 3+ источниках влияют на результат запроса.
- **[A1-R10.3] MUST NOT** Накладывать в `ГДЕ` отбор по полям правой таблицы после `ЛЕВОЕ СОЕДИНЕНИЕ`, если нужно сохранить семантику левого соединения; такой отбор размещать в условии соединения, параметрах виртуальной таблицы или предварительной выборке.
- **[A1-R10.4] SHOULD** Избегать соединений с вложенными запросами на рабочих объёмах; предпочитать предварительную выборку во временные таблицы.

### 1.5 Пустой результат

- **[A1-R11] MUST** Проверять пустой результат перед обработкой (ИТС #std438).

```bsl
Выборка = Запрос.Выполнить().Выбрать();
Если Выборка.Пустой() Тогда
    Возврат;
КонецЕсли;
```

### 1.6 Сортировка

- **[A1-R12] MUST NOT** Использовать `УПОРЯДОЧИТЬ ПО` без необходимости (ИТС #std412). Сортировка на уровне запроса — только когда данные потребляются напрямую пользователем.
- **[A1-R13] SHOULD** Использовать `АВТОУПОРЯДОЧИВАНИЕ` только когда действительно нужен платформенный порядок представления ссылочных данных или иерархии; в остальных случаях задавать порядок явно или не сортировать вовсе.

### 1.7 NULL и иерархия

- **[A1-R14] MUST** Различать `NULL`, пустую строку, `0`, `Неопределено` и пустую ссылку: это разные состояния данных.
- **[A1-R15] MUST** Проверять `NULL` через `ЕСТЬ NULL` или `ВЫБОР`, а не через сравнение с пустыми значениями.
- **[A1-R16] SHOULD** Для иерархических справочников использовать `В ИЕРАРХИИ`, когда нужен весь подчиненный узел, а не только непосредственный `Родитель`.
- **[A1-R17] SHOULD** Помнить, что `В ИЕРАРХИИ(&Узел)` включает в результат и сам переданный узел.

### 1.8 Агрегация и представления

- **[A1-R18] MUST** Фильтрацию по агрегатам выполнять в `ИМЕЮЩИЕ`, а не в `ГДЕ`.
- **[A1-R19] MUST** В `СГРУППИРОВАТЬ ПО` указывать имена полей, а не их псевдонимы.
- **[A1-R20] SHOULD** Использовать `ПРЕДСТАВЛЕНИЕ()` только для пользовательского отображения; не применять его в сравнениях, соединениях и вычисляемых условиях.

### 1.9 Временные таблицы

- **[A1-R21] SHOULD** Для больших списков, повторного использования набора данных и упрощения сложных запросов предпочитать временные таблицы (`ПОМЕСТИТЬ`) вместо вложенных запросов.
- **[A1-R22] SHOULD** Временные таблицы индексировать по полям, которые участвуют в отборах и соединениях (`ИНДЕКСИРОВАТЬ ПО`).

## 3. Запросы: оформление

### 3.1 Формат текста

- **[A3-R1] MUST** Следовать стандарту ИТС #std437:
  - Одна секция — один блок строк (пустая строка между секциями).
  - Алиасы выровнены по колонке `КАК`.
  - Все параметры — `&ИмяПараметра`.

```bsl
ТекстЗапроса =
"ВЫБРАТЬ
|   Т.Ссылка                                  КАК Ссылка,
|   Т.Наименование                            КАК Наименование,
|
|   СУММА(Т.СуммаДокумента)                   КАК СуммаДокумента
|
|ИЗ
|   Документ.РеализацияТоваровУслуг КАК Т
|
|ГДЕ
|   Т.Дата МЕЖДУ &ДатаНач И &ДатаКон
|
|СГРУППИРОВАТЬ ПО
|   Т.Ссылка,
|   Т.Наименование";
```

> **Примечание:** `УПОРЯДОЧИТЬ ПО` используется только когда данные потребляются напрямую пользователем (см. [A1-R12]). Сортировка в отчёте СКД настраивается средствами СКД, не в запросе.

- **[A4-R1] MUST** Если текст из запроса/СКД попадает в интерфейс — использовать `НСтр()`, не строковые литералы (ИТС #std762).

```bsl
// ✅ ПРАВИЛЬНО
```

## 5. СКД: принципы

### 5.1 Архитектура отчёта

- **[A5-R1] SHOULD** Один набор данных (запрос), пока не доказана необходимость нескольких.
- **[A5-R2] MUST** Группировки и ресурсы соответствуют постановке задачи.
- **[A5-R3] MUST** Расшифровка ведёт по ссылкам/ключам, не по текстовым представлениям.
- **[A5-R3.1] MUST** Для элементов структуры `Таблица` и `Диаграмма` определять ресурсы; без ресурсов такие отчёты считаются некорректно спроектированными.
- **[A5-R3.2] SHOULD** Использовать `Группировку` для линейного отчёта и списков, `Таблицу` — только когда действительно нужны строки/колонки и ресурсы на пересечении.
- **[A5-R3.3] SHOULD** Один отчет проектировать как одну схему компоновки с несколькими вариантами, если отличаются только представление, поля, сортировки, отборы или оформление.

### 5.2 Параметры

- **[A5-R4] MUST** Пользовательские параметры — явные, с понятными дефолтами.
- **[A5-R5] SHOULD** Технические параметры — скрытые, задаваемые программно.
- **[A5-R5.1] SHOULD** Для параметров периода задавать тип с составом даты `Дата`, а конец периода при необходимости вычислять через `КонецПериода(..., "День")`.
- **[A5-R5.2] MUST** Если параметр участвует в получении данных всегда, задавать для него использование `Всегда`.
- **[A5-R5.3] SHOULD** Быстрые пользовательские настройки включать только для действительно частых действий пользователя; остальные оставлять в полной форме настроек.

### 5.3 Варианты

- **[A5-R6] SHOULD** Минимум 1 основной вариант. Отдельные варианты — по ролям (бухгалтер, менеджер, руководитель).
- **[A5-R6.1] SHOULD** Не плодить отдельные отчеты, если достаточно нового варианта текущего отчета.
- **[A5-R6.2] SHOULD** Помнить, что у каждого варианта отчета свой набор пользовательских настроек.

### 5.4 Программное вмешательство

Допустимо когда:
- Значения параметров зависят от прикладной логики/прав.
- Нужно подменить набор данных по режиму.
- Нужно динамически менять заголовки/оформление.

**Правило:** вмешательство — централизованно в модуле отчёта (точки компоновки), не размазано по формам.

### 5.5 Несколько наборов данных и программная инициализация

- **[A5-R7] MUST** При связи наборов данных СКД учитывать, что это левое соединение; родительский набор выбирать осознанно, так как именно его строки гарантированно сохраняются.
- **[A5-R8] SHOULD** Если нужно вывести данные из нескольких источников без связи строк, использовать набор данных `Объединение`, а не связь наборов.
- **[A5-R9] SHOULD** Для внешних данных в памяти использовать набор данных `Объект` и явно описывать его поля.
- **[A5-R10] SHOULD** Для автозапуска отчета при открытии формы использовать параметр `СформироватьПриОткрытии`.
- **[A5-R11] MUST** Если значения по умолчанию задаются для быстрых пользовательских настроек, изменять их в `ПередЗагрузкойПользовательскихНастроекНаСервере`.
- **[A5-R12] MUST** Если значения по умолчанию задаются для настроек варианта отчета до применения пользовательских настроек, изменять их в `ПередЗагрузкойВариантаНаСервере`.
- **[A5-R13] SHOULD** В пользовательских полях СКД не рассчитывать на вызов функций общих модулей; использовать язык выражений компоновки и подготовленные поля набора данных.

## 6. СКД: типовой отчёт «Остатки + Обороты»

### 6.1 Параметры

| Параметр | Тип | Обязательный |
|----------|-----|-------------|
| ДатаНач | Дата | Да |
| ДатаКон | Дата | Да |
| Склад | СправочникСсылка.Склады | Нет |

### 6.2 Запрос

```bsl
"ВЫБРАТЬ
|   Д.Номенклатура                          КАК Номенклатура,
|   Д.Склад                                КАК Склад,
|   Д.КоличествоНачальныйОстаток            КАК НачОстаток,
|   Д.КоличествоПриход                      КАК Приход,
|   Д.КоличествоРасход                      КАК Расход,
|   Д.КоличествоКонечныйОстаток             КАК КонОстаток
|ИЗ РегистрНакопления.Товары.Обороты(&ДатаНач, &ДатаКон, , ) КАК Д
|ГДЕ (&Склад ЕСТЬ NULL ИЛИ Д.Склад = &Склад)"
```

### 6.3 Настройки СКД

- **Группировки:** Номенклатура → (опц.) Склад
- **Ресурсы:** НачОстаток, Приход, Расход, КонОстаток
- **Расшифровка:** по Номенклатуре/Складу → движения/документы

# PART B — КОД И ПЛАТФОРМА

## 7. Типы данных

### 7.1 Примитивные типы

| Тип | Пример | Примечание |
|-----|--------|-----------|
| Число | `1500`, `3.14` | Произвольная точность |
| Строка | `"Товар"`, `'Кавычки'` | Unicode, многострочные через `\|` |
| Дата | `Дата(2024,1,15)`, `'20240115'` | Точность до секунды |
| Булево | `Истина`, `Ложь` | — |
| Неопределено | `Неопределено` | Аналог null |
| УникальныйИдентификатор | `Новый УникальныйИдентификатор` | GUID |

### 7.2 Приведение типов

- **[B7-R1] MUST** Явно приводить типы при сравнении. Платформа не выдаст ошибку, но результат непредсказуем.

```bsl
// ✅ ПРАВИЛЬНО
Если Число(СтрокаЗначение) > 0 Тогда ...

// ⚠️ РИСК
Если СтрокаЗначение > 0 Тогда ...
```

### 7.3 Проверка на пустоту

```bsl
ПустаяСтрока(Строка);         // Строка
Значение = Неопределено;      // Любое
Ссылка.Пустая();              // Ссылка
ЗначениеЗаполнено(Значение);  // Универсально (БСП)
```

## 8. Операторы BSL

### 8.1 Условия

- **[B8-R1] MUST NOT** Использовать тяжёлые вычисления внутри `?()` — все 3 аргумента вычисляются до возврата.

```bsl
// ⚠️ Плохо: запрос выполнится в любом случае
Результат = ?(Условие, Запрос.Выполнить(), Неопределено);

// ✅ Хорошо
Если Условие Тогда
    Результат = Запрос.Выполнить();
Иначе
    Результат = Неопределено;
КонецЕсли;
```

### 8.2 Циклы

- **[B8-R2] SHOULD** Использовать `Для Каждого ... Из` вместо перебора по индексам.

```bsl
Для Каждого Строка Из ТЗ Цикл
    Если Строка.Пометка = Ложь Тогда
        Продолжить;
    КонецЕсли;
    Если Строка.Стоп Тогда
        Прервать;
    КонецЕсли;
КонецЦикла;
```

### 8.3 Процедуры и функции

- **[B8-R3] MUST NOT** Менять параметры внутри функции — антипаттерн.
- **[B8-R4] SHOULD** Использовать ранние `Возврат` вместо глубокой вложенности.

```bsl
// ✅ ПРАВИЛЬНО: ранний возврат
Функция РассчитатьСкидку(Сумма)
    Если Сумма <= 0 Тогда
        Возврат 0;
    КонецЕсли;
    Если Сумма > 10000 Тогда
        Возврат Сумма * 0.1;
    КонецЕсли;
    Возврат 0;
КонецФункции
```

### 8.4 Исключения

```bsl
Попытка
    Объект.Записать();
Исключение
    ЗаписьЖурналаРегистрации(
        "ЗаписьОбъекта",
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ОписаниеОшибки()
    );
    ВызватьИсключение;
КонецПопытки;
```

## 9. Коллекции значений

### 9.1 Выбор коллекции

- **[B9-R1] MUST** Выбирать коллекцию по назначению, а не по привычке. Не использовать `Соответствие` когда достаточно `Структура`.
- **[B9-R2] MUST** Использовать `Фиксированная*` коллекцию при передаче данных в клиентский код для защиты от модификации.

| Коллекция | Когда использовать |
|-----------|-------------------|
| **Массив** | Упорядоченный список, доступ по индексу |
| **Структура** | Передача параметров в метод (ключ = строка) |
| **Соответствие** | Ключ — любой тип (не только строка) |
| **ТаблицаЗначений** | Буфер между запросом и СКД/печатью |
| **ДеревоЗначений** | Иерархические данные в формах |
| **Фиксированная*** | Защита от модификации при передаче |

### 9.2 ТаблицаЗначений

- **[B9-R3] MUST** Указывать `ОписаниеТипов` для колонок ТЗ, создаваемой вручную (ИТС #std437).

```bsl
ТЗ = Новый ТаблицаЗначений;
ТЗ.Колонки.Добавить("Номенклатура", ОписаниеТипов("СправочникСсылка.Номенклатура"));
ТЗ.Колонки.Добавить("Количество", ОписаниеТипов("Число", 15, 3));

Строка = ТЗ.Добавить();
Строка.Номенклатура = Ссылка;
Строка.Количество = 10;

// Из запроса
ТЗ = Запрос.Выполнить().Выгрузить();

// Операции
ТЗ.Сортировать("Сумма Убыв");
ТЗ.Свернуть("Номенклатура", "Количество, Сумма");
Найденная = ТЗ.Найти(Ссылка, "Номенклатура");
```

## 10. Объектные типы: Ссылка → Объект → Менеджер

### 10.1 Три уровня

| Уровень | Что | Когда |
|---------|-----|-------|
| **Менеджер** | `Справочники.Номенклатура` | Создание, поиск, перебор |
| **Ссылка** | Лёгкий указатель (ID) | Отборы, сравнения, реквизиты |
| **Объект** | Полные данные + методы | Изменение и запись |

- **[B10-R1] MUST** Начинать со Ссылки. Переходить к Объекту только когда нужно изменить данные.

```bsl
// ✅ Ссылка — для чтения/отбора
Ссылка = Справочники.Номенклатура.НайтиПоКоду("001");
Сообщить(Ссылка.Наименование);

// ✅ Объект — для изменения
Объект = Ссылка.ПолучитьОбъект();
Объект.Наименование = "Новое имя";
Объект.Записать();
```

### 10.2 Модуль объекта vs Модуль формы

- **[B10-R2] MUST** В модуле объекта использовать `ЭтотОбъект`, в модуле формы — `Объект`.

| Контекст | Переменная |
|----------|-----------|
| Модуль объекта | `ЭтотОбъект` |
| Модуль формы | `Объект` |

## 10.5 Документирование кода

### 10.5.1 Docstring для процедур и функций

- **[B10.5-R1] MUST** Документировать каждую экспортную процедуру/функцию.
- **[B10.5-R2] MUST** Размещать комментарий ДО директивы компиляции (`&НаСервере`, `&НаКлиенте`).
- **[B10.5-R3] SHOULD** Документировать внутренние функции со сложной логикой.

```bsl
// Рассчитывает скидку на основании суммы заказа.
//
// Параметры:
//   СуммаЗаказа - Число - Общая сумма заказа
//   ТипКлиента  - СправочникСсылка.ТипыКлиентов - Тип клиента
//
// Возвращает:
//   Число - Размер скидки в процентах
//
// Пример:
//   Скидка = РассчитатьСкидку(15000, Премиум); // Вернёт 15
//
&НаСервереБезКонтекста
Функция РассчитатьСкидку(СуммаЗаказа, ТипКлиента) Экспорт
```

### 10.5.2 Obsolete-маркировка

- **[B10.5-R4] MUST** Помечать устаревшие методы комментарием `// Obsolete.` с указанием альтернативы.

```bsl
// Obsolete. Использовать РассчитатьСкидкуV2().
// Причина: не учитывает сезонные коэффициенты.
Функция РассчитатьСкидку(Сумма) Экспорт
```

### 10.5.3 Формат описания параметров

- **[B10.5-R5] MUST** Использовать формат: `Имя - Типы - Описание`.
- **[B10.5-R6] SHOULD** Для сложных типов указывать вложенные свойства через `*`, `**`, `***`.

```bsl
// Параметры:
//   Заказ     - ДокументСсылка.ЗаказПокупателя - Ссылка на заказ
//   * Товары - ТаблицаЗначений - Состав заказа
//   ** Номенклатура - СправочникСсылка.Номенклатура
//   ** Количество  - Число
```

## 11. Управляемые формы

### 11.1 Клиент/сервер

- **[B11-R1] MUST** Разделять клиентский и серверный код в модулях управляемых форм:
  - `&НаКлиенте` — UI-логика, вызов сервера.
  - `&НаСервере` / `&НаСервереБезКонтекста` — данные, вычисления.
- **[B11-R2] MUST** Минимизировать передачу данных клиент↔сервер.
- **[B11-R3] SHOULD** Использовать `&НаСервереБезКонтекста` где возможно (быстрее).

> **Примечание:** Для БСП-модулей см. [C16-R1] — суффиксы и флаги общих модулей.

```bsl
&НаКлиенте
Процедура КомандаВыполнить(Команда)
    Результат = СервернаяФункция(Параметр);
    Если Результат Тогда
        Оповестить("ДанныеОбновлены");
    КонецЕсли;
КонецПроцедуры

&НаСервереБезКонтекста
Функция СервернаяФункция(Параметр)
    Запрос = Новый Запрос(ТекстЗапроса);
    Запрос.УстановитьПараметр("Параметр", Параметр);
    Возврат Не Запрос.Выполнить().Пустой();
КонецФункции
```

### 11.2 Динамические списки

- **[B11-R4] SHOULD** Использовать динамические списки для больших объёмов (автоподкачка).
- **[B11-R5] SHOULD** Настраивать отборы на сервере через `ОбработкаПолученияДанных`.

## 12.5 Асинхронность (Async/Wait, Promise)

### 12.5.1 Асинхронные процедуры

- **[B12.5-R1] MUST** Использовать модификатор `Async` для процедур, выполняющих долгие операции (HTTP-запросы, запись файлов, тяжёлые вычисления).
- **[B12.5-R2] MUST** Использовать `Wait` только внутри `Async`-процедур.
- **[B12.5-R3] SHOULD** Возвращать `Promise` из функций, вызываемых асинхронно.

```bsl
// ✅ ПРАВИЛЬНО: асинхронный HTTP-запрос
&НаКлиенте
Асинхронная Процедура ЗагрузитьДанные(Команда)
    Результат = Ждать HTTPВыполнитьЗапросAsync("https://api.example.com/data");
    ОбработатьРезультат(Результат);
КонецПроцедуры
```

### 12.5.2 Promise-обёртки

```bsl
// ✅ Promise-обёртка для HTTP-запроса
&НаСервере
Функция HTTPВыполнитьЗапросAsync(URL)
    Возврат Promise.Создать(
        Процедура (Выполнить, Отказать)
            Попытка
                Соединение = Новый HTTPСоединение(URL);
                Ответ = Соединение.Получить(Новый HTTPЗапрос("/"));
                Если Ответ.КодСостояния = 200 Тогда
                    Выполнить(Ответ.ПолучитьТелоКакСтроку());
                Иначе
                    Отказать("HTTP " + Ответ.КодСостояния);
                КонецЕсли;
            Исключение
                Отказать(ОписаниеОшибки());
            КонецПопытки;
        КонецПроцедуры
    );
КонецФункции
```

### 12.5.3 Ограничения

- **[B12.5-R4] MUST NOT** Использовать `Wait` в интерактивных обработчиках без индикации прогресса.
- **[B12.5-R5] SHOULD** Обрабатывать исключения в `Promise.Создать()` через `Отказать()`.

## 12.6 Фоновые и регламентные задания

### 12.6.1 Фоновые задания

- **[B12.6-R1] MUST** Использовать фоновые задания для операций длительностью > 5 секунд.
- **[B12.6-R2] MUST** Обрабатывать статус завершения: `СостояниеФонаВания()` или `ОповещениеОЗавершении`.
- **[B12.6-R3] SHOULD** Реализовывать отмену фонового задания через общий флаг/хранилище.

```bsl
// ✅ Запуск фонового задания
&НаСервере
Процедура ЗапуститьФоновоеЗадание()
    Параметры = Новый Структура("Параметр1", Значение);
    ФЗ = Новый ФоновоеЗадание("ОбработкаВДономФоне", Параметры);

    // Ожидание завершения с таймаутом
    Таймаут = 300; // 5 минут
    Пока ФЗ.Статус = СтатусФонаВания.Выполняется Цикл
        Если Таймаут = 0 Тогда
            ФЗ.Отменить();
            ВызватьИсключение "Превышен таймаут фонового задания";
        КонецЕсли;
        Таймаут = Таймаут - 1;
        Ждать 1;
    КонецЦикла;

    Если ФЗ.Статус = СтатусФонаВания.Ошибка Тогда
        ВызватьИсключение ФЗ.ОписаниеОшибки;
    КонецЕсли;
КонецПроцедуры
```

### 12.6.2 Регламентные задания

- **[B12.6-R4] SHOULD** Настраивать расписание с учётом пиковых нагрузок (ночное время для тяжёлых операций).
- **[B12.6-R5] MUST** Использовать `БлокировкаРегламентногоЗадания` для предотвращения параллельного запуска.
- **[B12.6-R6] SHOULD** Логировать начало и конец выполнения в журнал регистрации.

```bsl
// ✅ Регламентное задание с блокировкой
Процедура РегламентноеОбновлениеКурсов()
    Блокировка = БлокировкаСтандартныхПодсистем.БлокировкаРегламентногоЗадания("ОбновлениеКурсов", 600);
    Если Блокировка = Неопределено Тогда
        ЗаписьЖурналаРегистрации("ОбновлениеКурсов", УровеньЖурналаРегистрации.Информация, , , "Уже выполняется");
        Возврат;
    КонецЕсли;

    Попытка
        ЗаписьЖурналаРегистрации("ОбновлениеКурсов", УровеньЖурналаРегистрации.Информация, , , "Начало");
        // ... логика обновления ...
        ЗаписьЖурналаРегистрации("ОбновлениеКурсов", УровеньЖурналаРегистрации.Информация, , , "Завершение");
    Исключение
        ЗаписьЖурналаРегистрации("ОбновлениеКурсов", УровеньЖурналаРегистрации.Ошибка, , , ОписаниеОшибки());
    КонецПопытки;
КонецПроцедуры
```

### 12.6.3 Индикация прогресса

- **[B12.6-R7] SHOULD** Обновлять прогресс через `УстановитьПрогресс()` в хранилище.
- **[B12.6-R8] SHOULD** Реализовывать проверку флага отмены в цикле обработки.

```bsl
// ✅ Цикл с прогрессом и проверкой отмены
Всего = Массив.Количество();
Для Каждого Элемент Из Массив Цикл
    // Проверка отмены
    Если ХранилищеЗагрузки.Получить("Отмена") = Истина Тогда
        Прервать;
    КонецЕсли;

    // Обработка...

    // Обновление прогресса
    Прогресс = Цел(Номер / Всего * 100);
    ХранилищеЗагрузки.Сохранить("Прогресс", Прогресс);
КонецЦикла;
```

## 13.5 Кэширование

### 13.5.1 Стратегии кэширования

- **[B13.5-R1] SHOULD** Кэшировать результаты тяжёлых запросов, не изменяющихся в рамках сеанса (справочники, перечисления, виды цен).
- **[B13.5-R2] MUST** Инвалидировать кэш при изменении кэшируемых данных.
- **[B13.5-R3] SHOULD** Использовать `ХранилищеОбщихНастроек` для кэша между сеансами.

### 13.5.2 Паттерн Cache-Aside

```bsl
// ✅ Кэш справочника валют
&НаСервереБезКонтекста
Функция ПолучитьВалюту(Код)
    Ключ = "Валюта_" + Код;
    Кэш = ХранилищеОбщихНастроек.Загрузить(Ключ);
    Если Кэш <> Неопределено Тогда
        Возврат Кэш;
    КонецЕсли;

    Валюта = Справочники.Валюты.НайтиПоКоду(Код);
    Если Валюта.Пустая() Тогда
        Возврат Неопределено;
    КонецЕсли;

    Результат = Новый Структура("Ссылка, Наименование, Код", Валюта, Валюта.Наименование, Код);
    ХранилищеОбщихНастроек.Сохранить(Ключ, Результат, 3600); // TTL = 1 час
    Возврат Результат;
КонецФункции
```

### 13.5.3 Кэш на клиенте

- **[B13.5-R4] SHOULD** Кэшировать данные на клиенте через `НастройкиПрограммы` для часто используемых списков (статусы, виды операций).

## 13.6 Блокировки и разрешение конфликтов

### 13.6.1 Optimistic Locking

- **[B13.6-R1] SHOULD** Использовать `НомерВерсии` для оптимистичной блокировки при конкурентном редактировании.

```bsl
// ✅ Проверка версии перед записью
&НаСервере
Процедура ЗаписатьСИзменением(ОбъектДанных, Отказ)
    ТекущаяВерсия = ОбъектДанных.НомерВерсии;

    // Перечитать актуальные данные
    Актуальный = ОбъектДанных.Ссылка.ПолучитьОбъект();
    Если Актуальный.НомерВерсии <> ТекущаяВерсия Тогда
        Отказ = Истина;

        Возврат;
    КонецЕсли;

    ОбъектДанных.Записать();
КонецПроцедуры
```

### 13.6.2 Retry при конфликте блокировок

```bsl
        КонецЕсли;

```

## 19.5 Безопасность

### 19.5.1 SQL-инъекции и параметризация

- **[C19.5-R1] MUST** Все динамические значения в запросах передавать через параметры `&Имя` (см. [A1-R3]).
- **[C19.5-R2] MUST NOT** Использовать конкатенацию строк для построения условий запроса.

```bsl
// ❌ КРИТИЧЕСКАЯ УЯЗВИМОСТЬ: SQL-инъекция
Текст = "ВЫБРАТЬ * ИЗ Справочник.Контрагенты ГДЕ Наименование ПОДОБНО '" + Значение + "'";

// ✅ ПРАВИЛЬНО: параметризация
Запрос.Текст = "ВЫБРАТЬ * ИЗ Справочник.Контрагенты ГДЕ Наименование ПОДОБНО &Значение";
Запрос.УстановитьПараметр("Значение", "%" + Значение + "%");
```

### 19.5.2 Криптография

- **[C19.5-R3] SHOULD** Использовать `МенеджерКриптографии` для шифрования/подписи данных (ИТС #std441).
- **[C19.5-R4] MUST** Хранить сертификаты в защищённом хранилище, не в коде.

```bsl
// ✅ Шифрование данных
Менеджер = Новый МенеджерКриптографии;
Сертификат = Менеджер.ПолучитьСертификаты(Отбор)[0];
Зашифрованные = Менеджер.Зашифровать(Данные, Сертификат);

// ✅ Электронная подпись
Подпись = Менеджер.Подписать(Данные, Сертификат);
Если Менеджер.ПроверитьПодпись(Данные, Подпись, Сертификат) Тогда
    // Подпись верна
КонецЕсли;
```

### 19.5.3 Защита от загрузки внешних обработок

- **[C19.5-R5] MUST** Проверять цифровую подпись внешних обработок (`.epf`, `.erf`) перед загрузкой.
- **[C19.5-R6] SHOULD** Использовать белые списки допустимых обработок.

```bsl
// ✅ Проверка подписи внешней обработки
Обработка = ВнешниеОбработки.Создать(Путь, Ложь);
Если Не Обработка.ПодписьУстановлена() Тогда
    ВызватьИсключение "Обработка не подписана";
КонецЕсли;
```

### 19.5.4 Безопасность HTTP-сервисов

- **[C19.5-R7] MUST** Использовать HTTPS для всех внешних интеграций.
- **[C19.5-R8] SHOULD** Реализовывать авторизацию через токены (Bearer), не через логин/пароль в каждом запросе.

```bsl
// ✅ Bearer-авторизация в HTTP-сервисе
Функция ПроверитьТокен(Запрос)
    Токен = Запрос.Заголовки.Получить("Authorization");
    Если Токен = Неопределено ИЛИ Не ПроверитьТокенВБазе(Токен) Тогда
        Ответ = Новый HTTPСервисОтвет(401);
        Ответ.УстановитьТелоИзСтроки("Unauthorized");
        Возврат Ответ;
    КонецЕсли;
    Возврат Неопределено; // OK
КонецФункции
```

## 21. Интеграции: HTTP, OData, COM

### 21.1 HTTP-сервисы

- **[C21-R1] MUST** Возвращать корректные HTTP-коды (200, 201, 400, 404, 500).
- **[C21-R2] SHOULD** Использовать JSON для тела запроса/ответа.
- **[C21-R3] SHOULD** Настраивать CORS на уровне веб-сервера (Apache/Nginx) при кросс-доменных запросах из браузера.
- **[C21-R3.1] SHOULD** Явно устанавливать `Content-Type` и кодировку ответа, обычно `application/json` и `UTF-8`.
- **[C21-R3.2] SHOULD** Идентификаторы ресурса передавать в URL, а фильтры и режимы выборки — в параметрах запроса.

```bsl

    Возврат Ответ;
КонецФункции
```

### 21.2 HTTP-клиент

```bsl
Соединение = Новый HTTPСоединение("api.example.com");

    Данные = ПрочитатьJSON(Ответ.ПолучитьТелоКакСтроку());
```

- **[C21-R3.3] SHOULD** При серии обращений к одному сервису переиспользовать `HTTPСоединение`, а не создавать новое соединение на каждый вызов.
- **[C21-R3.4] SHOULD** Разбирать тело ответа только после проверки `КодСостояния`; ошибочные ответы логировать с кодом и кратким содержимым.

### 21.2.1 JSON

- **[C21-R3.5] SHOULD** Для больших JSON-документов использовать потоковые `ЧтениеJSON`/`ЗаписьJSON`; для небольших — `ПрочитатьJSON()`/`ЗаписатьJSON()`.

- **[C21-R3.8] SHOULD** Формат даты в JSON выбирать явно и согласовывать с принимающей стороной; по умолчанию предпочтителен ISO-формат.

## 23. XML-генерация метаданных: критические правила

### 23.1 InternalInfo

- **[C23-R1] MUST** `InternalInfo` обязателен для `Catalog`, `Document`, `InformationRegister`, `AccumulationRegister`.
- **[C23-R2] MUST NOT** `InternalInfo` запрещён для `Form`, `Subsystem`.
- **[C23-R3] MUST** Использовать `xr:ValueId` (не `xr:ObjectId`).

### 23.2 Формы

- **[C23-R4] MUST NOT** Добавлять `ExtendedPresentation`, `Explanation` в `Form`.
- **[C23-R5] MUST** Namespace: `http://v8.1c.ru/8.3/MDClasses`.

### 23.3 Табличные части

- **[C23-R6] MUST** `InternalInfo` с `CatalogTabularSection` + `CatalogTabularSectionRow`.
- **[C23-R7] MUST NOT** Атрибуты ТЧ: только `Name`, `Synonym`, `Comment`, `Type`. Без `FillValue`, `Indexing`.

### 23.4 Предопределённые элементы

- **[C23-R8] MUST** Хранить в `Ext/Predefined.xml`, не в основном XML.

### 23.5 ChildObjects

- **[C23-R9] MUST** `ChildObjects` обязателен для `ChartOfCharacteristicTypes` (даже пустой).
- **[C23-R10] MUST** `<Form>` внутри `<ChildObjects>`, до закрывающего тега.

### 23.6 UUID

- **[C23-R11] MUST** Генерировать уникальные UUID (`[guid]::NewGuid()`), не заглушки.

```yaml
# .gitlab-ci.yml
stages:

# Статический анализ (OneScript + SonarQube / 1С:CodeStyle)

  stage: analyze
  script:
    - os script:check_codestyle.os

vanessa-tests:
  stage: test

    - |
      $v8 = "${V8_PATH}\1cv8.exe"

        /Out "${CI_PROJECT_DIR}\test-results.xml"

        exit 1
      }
  artifacts:
    reports:
      junit: test-results.xml
    paths:
      - test-results.xml
    when: always

# Сборка .cf файла
build-cf:
  stage: build
  script:
    - |
      $v8 = "${V8_PATH}\1cv8.exe"
      & $v8 CONFIG /F "${CI_PROJECT_DIR}\src" /Out "${CI_PROJECT_DIR}\build.log"

      if (-not $cf) { Write-Error ".cf файл не создан"; exit 1 }
      Copy-Item $cf.FullName "${CI_PROJECT_DIR}\artifacts\$(Get-Date -Format 'yyyyMMdd_HHmmss').cf"
  artifacts:
    paths:
      - artifacts/*.cf
    expire_in: 30 days
  only:
    - main
    - release/*

# Развёртывание на тестовый сервер
deploy-test:

      $conn = New-Object -ComObject "V83.COMConnector"
```

```groovy
                 subject: "CI/CD FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                 body: "Check console: ${env.BUILD_URL}console"
        }
    }
}
```

> **⛔ БЛОКИРУЮЩЕЕ ПРАВИЛО: AI НЕ ДОЛЖЕН выдавать код, если хотя бы один пункт применимого чек-листа не пройден. Это не рекомендация — это жёсткий запрет. Код с ошибкой чек-листа = мусор, он не должен покидать AI.**

## 25. Чек-листы валидации

## 30. Технологический журнал

### 30.1 Настройка logcfg.xml

- **[E30-R1] SHOULD** Настраивать `logcfg.xml` на сервере 1С для диагностики проблем.
- **[E30-R2] MUST** Использовать уровни важности: `ERROR`, `WARNING`, `INFO`, `DEBUG`, `TRACE` (8.3.25 — 5 уровней).

```xml
<!-- ✅ Минимальная конфигурация logcfg.xml -->
<config xmlns="http://v8.1c.ru/v8/techlog">
  <log history="24" location="C:\1C\Logs">
    <event>
      <eq property="name" value="EXCP"/>
      <eq property="level" value="error"/>
    </event>
    <property value="excptext;desc"/>
  </log>
  <log history="24" location="C:\1C\Logs\Dbg">
    <event>
      <eq property="name" value="DBPOSTGRS"/>
      <eq property="level" value="debug"/>
    </event>
    <property value="sqltext;duration"/>
  </log>
</config>
```

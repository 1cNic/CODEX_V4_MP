# Журнал Ошибок И Выводов

Этот файл нужен как локальная память по ошибкам, уже пойманным в проекте.
Перед следующими изменениями по 1С нужно сверяться с ним.

## 1. Подчиненный справочник

- Ошибка: пытаться задавать владельца только через `Owners` без полного шаблона типовой выгрузки.
- Правильно: для подчиненного справочника сверяться с типовыми XML и добавлять не только `Owners`, но и `StandardAttributes` с `Owner`.
- Вывод: структурные свойства метаданных в 1С нельзя додумывать по памяти, только по живому эталону.

## 2. Русские имена в XML

- Ошибка: вставлять русские имена метаданных так, что в файле появляется перекодированная каша.
- Правильно: если есть риск проблем с кодировкой, использовать точный эталон или XML escape-последовательности для QName/имен в типах.
- Вывод: все ссылочные типы вида `cfg:CatalogRef...` и `cfg:Document...` нужно проверять особенно внимательно.

## 3. Внутреннее имя объекта и файл

- Ошибка: быстро переименовывать объект, но оставлять несовместимую структуру служебных папок/файлов.
- Правильно: при переименовании объекта проверять:
  - имя в `Configuration.xml`
  - основной XML объекта
  - папки `Forms`
  - папки `Commands`
  - ссылки в модулях и формах
- Вывод: переименование объекта в 1С всегда делать как целостную операцию.

## 4. Автоматическое действие при открытии формы

- Ошибка: запускать бизнес-логику в `ПриОткрытии`, если по сценарию пользователь должен нажать кнопку.
- Правильно: действие генерации вешать на явную команду/кнопку формы.
- Вывод: `ПриОткрытии` использовать только для инициализации интерфейса, а не для изменения данных, если это не согласовано явно.

## 5. Кнопки в управляемой форме

- Ошибка: надеяться, что команда сама появится в интерфейсе только из блока `Commands`.
- Правильно: если нужна гарантированно видимая кнопка, добавлять явный `CommandBar` и явный `Button` с `CommandName`.
- Вывод: в спорных местах UI лучше делать явную разметку формы, а не полагаться на автопанели.

## 6. Случайность в 1С

- Ошибка: использовать несуществующие функции вроде `СлучайноеЧисло` или сомнительные выражения без проверки платформы.
- Правильно: использовать только проверенные механизмы платформы и по возможности упрощать логику.
- Вывод: если задача тестовая, лучше простая детерминированная или проверенно работающая схема, чем “умная” случайность без верификации.

## 7. Не придумывать типовые паттерны

- Ошибка: сначала строить форму/обработку по предположению, а не по реальному примеру из типовой.
- Правильно: если пользователь просит “посмотри реальные примеры и практики”, сначала снять точный эталон, потом делать свою реализацию.
- Вывод: в 1С типовой пример почти всегда надежнее импровизации.

## 8. Что проверять перед передачей на загрузку

- Для новых объектов обязательно перепроверять:
  - есть ли ссылка в `Configuration.xml`
  - совпадает ли имя объекта и путь к файлу
  - не остались ли хвосты от старого имени
  - нет ли автозапуска логики из формы
  - нет ли обращения к несуществующим функциям платформы

## 9. Контроль остатков в учебной конфигурации

- Ошибка: строить контроль остатков через усложненный запрос с передачей `ТаблицаЗначений` в запрос или через лишнюю комбинацию виртуальной таблицы остатков и корректировок документа.
- Симптомы:
  - ошибка про индексируемое поле составного типа;
  - нестабильное поведение при проведении расхода;
  - вместо штатного отказа появляется общая ошибка проведения.
- Правильно: для этой учебной конфигурации использовать максимально простую и прозрачную схему:
  - агрегировать строки документа в BSL;
  - считать остаток отдельным параметризованным запросом по движениям регистра;
  - исключать из расчета текущий документ через `Регистратор <> &Ссылка`.
- Вывод: в простом тестовом решении 1С контроль остатков лучше делать проще и прозрачнее, а не через тяжелую универсальную схему.

## 10. Текст ошибки проведения для пользователя

- Ошибка: ограничиваться `Сообщить()` в модуле объекта документа, если пользователь должен увидеть понятный текст именно в системном окне ошибки проведения.
- Симптом: внизу формы появляется нормальное сообщение, но в основном окне ошибки пользователь видит только общее `Не удалось провести...` или `[ОшибкаХранимыхДанных]`.
- Правильно: если нужен понятный текст именно в системном окне, формировать его явно и поднимать через `ВызватьИсключение` с пользовательским текстом.
- Вывод: `Сообщить()` хорошо подходит для панели сообщений, но не заменяет явный текст исключения при отказе проведения.

## 11. Менеджеры метаданных в клиентском модуле формы

- Ошибка: обращаться из `&НаКлиенте` к `Справочники.Характеристики.ПустаяСсылка()` без подтверждения, что такой контекст доступен в модуле формы.
- Симптом: при инициализации модуля формы появляется ошибка `Переменная не определена (Справочники)`.
- Правильно: в учебной управляемой форме всю работу с пустой ссылкой и подбором ссылочных значений выносить на `&НаСервере`, а на клиенте только присваивать готовый результат.
- Вывод: даже для простой автоподстановки не нужно тянуть менеджеры метаданных в клиентский код, если их доступность в этом контексте не подтверждена локальным эталоном.

## 12. UsualGroup в XML формы

- Ошибка: задавать для `UsualGroup` неподтвержденные значения свойств без локального рабочего эталона.
- Симптом: при загрузке формы возникает XDTO-ошибка `Несоответствие свойства и элемента данных XDTO: Свойство: 'Group'`.
- Правильно: брать набор свойств `UsualGroup` только из уже загружаемых форм проекта. Для этого проекта подтвержденный шаблон: `Group = Vertical`, `Behavior = Usual`, `Representation = None`.
- Вывод: даже похожие значения свойств формы нельзя подставлять по памяти, если они не подтверждены точным локальным примером.

## 13. Параметры программной СКД

- Ошибка: вызывать `Настройки.ПараметрыДанных.УстановитьЗначениеПараметра("ИмяПараметра", ...)`, если параметр не объявлен в самой схеме компоновки данных.
- Симптом: ошибка выполнения `Параметр с указанным именем не найден`.
- Правильно: либо явно заводить параметры в схеме по точному эталону, либо для простой учебной конфигурации не использовать виртуальные параметры СКД и строить запрос без них.
- Вывод: в программной СКД имя параметра само по себе ничего не создаёт; сначала должен существовать параметр схемы, и только потом ему можно присваивать значение.
## 14. ДАТАВРЕМЯ в тексте запроса

- Ошибка: собирать аргументы `ДАТАВРЕМЯ(...)` через `Формат(..., "ЧГ=0")` без подтвержденного эталона.
- Симптом: в тексте запроса отдельные компоненты даты могут превратиться в пустое значение, и запрос падает с ошибкой вида `Ожидается выражение ","`.
- Правильно: для простой сборки даты в тексте запроса использовать `Строка(Год(...))`, `Строка(Месяц(...))` и так далее.
- Вывод: в учебной конфигурации простая конкатенация числовых частей даты надежнее, чем форматные маски, поведение которых не подтверждено локальным примером.

## 15. Строка(Число) в тексте запроса

- Ошибка: вставлять в текст запроса результат `Строка(Число)` без очистки.
- Симптом: внутри числа может появиться пробел или неразрывный пробел, например в годе `2 026`, и запрос падает с ошибкой синтаксиса.
- Правильно: перед вставкой чисел в текст запроса очищать строку через `СтрЗаменить(..., Символы.НПП, "")` и `СтрЗаменить(..., " ", "")`.
- Вывод: даже простое преобразование числа в строку в 1С нельзя считать безопасным для текста запроса без очистки разделителей разрядов.

## 16. Не собирать учебный отчет на самодельной программной СКД без макета

- Ошибка: пытаться с нуля программно собрать СКД-отчет, если в проекте нет подтвержденного шаблона макета и нет точного локального примера нужной структуры.
- Симптом: каскад ошибок XDTO, параметров, запроса и в итоге пустой или непонятный пользователю результат.
- Правильно: для учебной конфигурации использовать локально подтвержденный паттерн из книги — форма отчета с параметрами и результатом запроса в табличном представлении, либо отчет с реальным макетом `ТабличныйДокумент`.
- Вывод: в учебном проекте лучше простой и устойчивый отчет, чем формально “более правильная” СКД без надежного локального эталона.
## 17. ПРЕДСТАВЛЕНИЕ и УПОРЯДОЧИТЬ в запросе отчета

- Ошибка: сортировать результат запроса по полям, полученным через `ПРЕДСТАВЛЕНИЕ(...)`, прямо в предложении `УПОРЯДОЧИТЬ`.
- Симптом: ошибка запроса вида `Операция не разрешена в предложении "УПОРЯДОЧИТЬ"`.
- Правильно: использовать `ПРЕДСТАВЛЕНИЕ(...)` только для вывода, а сортировку выполнять после `Выгрузить()` через `ТаблицаЗначений.Сортировать(...)`.
- Вывод: в учебном отчете безопаснее сначала получить данные для показа, а декоративную сортировку выполнять уже в BSL, а не усложнять текст запроса.

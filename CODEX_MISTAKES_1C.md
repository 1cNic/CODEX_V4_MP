# Журнал Ошибок И Выводов

Этот файл нужен как локальная память по ошибкам, уже пойманным в проекте.
Перед следующими изменениями по 1С нужно сверяться с ним.

## 1. Подчиненный справочник

- Ошибка: пытаться задавать владельца только через `Owners` без полного шаблона типовой выгрузки.
- Правильно: для подчиненного справочника сверяться с типовыми XML и добавлять не только `Owners`, но и `StandardAttributes` с `Owner`.
- Вывод: структурные свойства метаданных в 1С нельзя додумывать по памяти, только по живому эталону.

## 2. Русские имена в XML

- Ошибка: вставлять русские имена метаданных так, что в файле появляется перекодированная каша.
- Правильно: если есть риск проблем с кодировкой, использовать точный эталон или XML escape-последовательности для QName/имен в типах.
- Вывод: все ссылочные типы вида `cfg:CatalogRef...` и `cfg:Document...` нужно проверять особенно внимательно.

## 3. Внутреннее имя объекта и файл

- Ошибка: быстро переименовывать объект, но оставлять несовместимую структуру служебных папок/файлов.
- Правильно: при переименовании объекта проверять:
  - имя в `Configuration.xml`
  - основной XML объекта
  - папки `Forms`
  - папки `Commands`
  - ссылки в модулях и формах
- Вывод: переименование объекта в 1С всегда делать как целостную операцию.

## 4. Автоматическое действие при открытии формы

- Ошибка: запускать бизнес-логику в `ПриОткрытии`, если по сценарию пользователь должен нажать кнопку.
- Правильно: действие генерации вешать на явную команду/кнопку формы.
- Вывод: `ПриОткрытии` использовать только для инициализации интерфейса, а не для изменения данных, если это не согласовано явно.

## 5. Кнопки в управляемой форме

- Ошибка: надеяться, что команда сама появится в интерфейсе только из блока `Commands`.
- Правильно: если нужна гарантированно видимая кнопка, добавлять явный `CommandBar` и явный `Button` с `CommandName`.
- Вывод: в спорных местах UI лучше делать явную разметку формы, а не полагаться на автопанели.

## 6. Случайность в 1С

- Ошибка: использовать несуществующие функции вроде `СлучайноеЧисло` или сомнительные выражения без проверки платформы.
- Правильно: использовать только проверенные механизмы платформы и по возможности упрощать логику.
- Вывод: если задача тестовая, лучше простая детерминированная или проверенно работающая схема, чем “умная” случайность без верификации.

## 7. Не придумывать типовые паттерны

- Ошибка: сначала строить форму/обработку по предположению, а не по реальному примеру из типовой.
- Правильно: если пользователь просит “посмотри реальные примеры и практики”, сначала снять точный эталон, потом делать свою реализацию.
- Вывод: в 1С типовой пример почти всегда надежнее импровизации.

## 8. Что проверять перед передачей на загрузку

- Для новых объектов обязательно перепроверять:
  - есть ли ссылка в `Configuration.xml`
  - совпадает ли имя объекта и путь к файлу
  - не остались ли хвосты от старого имени
  - нет ли автозапуска логики из формы
  - нет ли обращения к несуществующим функциям платформы

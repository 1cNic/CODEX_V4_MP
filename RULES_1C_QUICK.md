# RULES 1C CORE

Короткий обязательный стандарт для ежедневной разработки на `1С:Предприятие 8.3`.

Этот файл является практическим ядром `RULES_1C.md`.
Если нужен полный контекст, редкие случаи, большие примеры и расширенные пояснения, смотри `RULES_1C.md`.

## 1. Обязательный порядок работы

Перед любыми изменениями по 1С:
1. Сначала сверяйся с `CODEX_MISTAKES_1C.md`.
2. Потом смотри `PATTERNS_1C.md`, если в проекте уже есть подходящий успешный пример.
3. Потом ищи локальный эталон в текущем проекте, `samples_1c` или `tools/reference_indexes`.
4. После этого делай одну реализацию без архитектурных прыжков.
5. Если точного эталона нет, честно остановись и скажи, что нужен образец, а не сочиняй.

## 2. Что запрещено

1. Не выдумывай XML-структуру метаданных.
2. Не выдумывай управляемые формы, события формы, имена форм и параметры `ОткрытьФорму(...)`.
3. Не выдумывай `cfg:CatalogRef...`, `cfg:CatalogObject...`, `cfg:Document...`, `cfg:EnumRef...`.
4. Не выдумывай `WindowOpeningMode`, `Attributes`, `Commands`, `ChildItems`, `StandardAttributes`.
5. Не подменяй стандартное поведение формы пользовательским кодом без подтвержденного локального примера.
6. Не считай смену текста ошибки решением, пока не проверена корневая причина.

## 3. Перед началом

1. Если требования задачи размыты, сначала уточни их.
2. Используй правила именования и суффиксы БСП.
3. Если правило стандарта конфликтует с задачей, явно зафиксируй конфликт и предложи безопасную альтернативу.
4. Для учебной конфигурации предпочитай простое и прозрачное решение, а не тяжелую “универсальную” архитектуру.

## 4. Запросы

1. Не используй `ВЫБРАТЬ *`.
2. Все поля перечисляй явно и задавай алиасы через `КАК`.
3. Все значения передавай только через параметры `&ИмяПараметра`.
4. Устанавливай параметры через `Запрос.УстановитьПараметр()`.
5. Отборы располагай как можно ближе к источнику данных.
6. Не применяй функции к полям в `ГДЕ`, если можно отобрать по диапазону или индексируемому значению.
7. Не сортируй в запросе без реальной необходимости.
8. Перед обработкой результата проверяй, что выборка не пустая.
9. После `ЛЕВОЕ СОЕДИНЕНИЕ` не фильтруй поля правой таблицы в `ГДЕ`, если хочешь сохранить все строки левого источника.
10. Для фильтрации по агрегатам используй `ИМЕЮЩИЕ`, а не `ГДЕ`.
11. В `СГРУППИРОВАТЬ ПО` указывай реальные имена полей, не псевдонимы.
12. Различай `NULL`, пустую строку, пустую ссылку и `Неопределено`; `NULL` проверяй через `ЕСТЬ NULL`.
13. `ПРЕДСТАВЛЕНИЕ()` используй только для вывода пользователю, не для сравнений, отборов и сортировки в запросе.
14. Для иерархии используй `В ИЕРАРХИИ`, если нужен весь подчинённый узел, и помни, что сам узел тоже войдёт в результат.
15. Для сложных или повторно используемых выборок предпочитай временные таблицы `ПОМЕСТИТЬ` и индексируй их через `ИНДЕКСИРОВАТЬ ПО`.

## 5. СКД и отчеты

1. Для `Таблицы` и `Диаграммы` обязательно определяй ресурсы; для простого списка достаточно `Группировки`.
2. Если меняются только поля, оформление, сортировка, отборы или представление, делай новый вариант отчета, а не новый отчет.
3. У каждого варианта отчета свои пользовательские настройки; учитывай это при доработках.
4. Параметры, которые всегда участвуют в получении данных, ставь в использование `Всегда`.
5. Для периода обычно задавай тип даты без времени и вычисляй конец дня через `КонецПериода(..., "День")`.
6. В быстрые пользовательские настройки выноси только самые частые действия; остальное оставляй в полной настройке.
7. Связи наборов данных в СКД считай левым соединением и внимательно выбирай родительский набор.
8. Если источники не нужно связывать построчно, используй набор данных `Объединение`.
9. Для внешних данных из памяти используй набор данных `Объект` и явно описывай его поля.
10. Не собирай программную СКД с нуля без подтвержденного локального шаблона; для учебной задачи допустим простой отчет на форме, если это честно соответствует ТЗ.

## 6. Код BSL

1. Код должен быть готов к вставке, без заглушек и комментариев вида «тут ваш код».
2. Повторно используемую логику выноси в общие модули.
3. Не создавай большие универсальные `god`-модули.
4. Экспортируй только действительно нужные процедуры и функции.
5. Начинай работу со `Ссылки`; к `Объекту` переходи только когда действительно нужно менять данные.
6. В модуле объекта используй `ЭтотОбъект`, в модуле формы `Объект`.
7. Документируй экспортные процедуры и функции; комментарий размещай до директивы компиляции.
8. Если поведение кода непонятно, сначала воспроизведи его в отладке, а не додумывай.
9. Для быстрых изолированных экспериментов удобно использовать внешние обработки и отчеты.

## 7. Формы, клиент и сервер

1. UI и обработчики формы держи на клиенте.
2. Запросы, проведение, расчёты и работу с данными выноси на сервер.
3. Явно указывай директивы `&НаКлиенте`, `&НаСервере`, `&НаСервереБезКонтекста`.
4. Если контекст формы не нужен, предпочитай `&НаСервереБезКонтекста`.
5. Минимизируй передачу данных между клиентом и сервером.
6. Предпочитай немодальный подход в управляемом приложении.
7. Долгие операции делай асинхронно; всё, что дольше 5 секунд, выноси в фоновые задания.
8. Для действий, которые должны запускаться по кнопке, не размещай бизнес-логику в `ПриОткрытии`; используй явную команду формы и явную кнопку.
9. Если нужна гарантированно видимая кнопка, не надейся только на автопанель команд; добавляй её явно по локальному эталону.

## 8. Транзакции и производительность

1. Транзакцию открывай как можно позже и закрывай как можно раньше.
2. Не вызывай интерактивные методы внутри транзакции.
3. Внутри транзакций используй `Попытка...Исключение`.
4. Для больших выборок и сервисов используй пагинацию.
5. При кэшировании обязательно продумывай инвалидирование кэша.

## 9. Регистры и объекты

1. Измерения регистра это то, по чему действительно нужен срез или группировка.
2. Ресурсы это то, что агрегируется и рассчитывается.
3. Не добавляй измерения, по которым движение бывает только в одну сторону; такие данные делай реквизитами.
4. При перепроведении используй `Движения.Очистить()`.
5. Для учебного контроля остатков предпочитай простую и прозрачную схему, а не тяжелую универсальную.

## 10. Безопасность и интеграции

1. Не храни пароли, сертификаты и секреты в открытом виде.
2. Используй `ЗащищённоеХранилище` и безопасные механизмы БСП.
3. Для внешних интеграций используй HTTPS.
4. В HTTP-сервисах возвращай корректные коды ответа.
5. В HTTP-ответах явно задавай `Content-Type` и кодировку, обычно `application/json` + `UTF-8`.
6. При нескольких вызовах одного сервиса переиспользуй `HTTPСоединение`.
7. Тело HTTP-ответа разбирай только после проверки `КодСостояния`.
8. Для больших JSON используй потоковые `ЧтениеJSON` и `ЗаписьJSON`; для небольших хватит `ПрочитатьJSON()` и `ЗаписатьJSON()`.
9. Не отключай `ПроверятьСтруктуру` у `ЗаписьJSON` без явной причины.
10. Формат даты в JSON выбирай явно; по умолчанию предпочитай ISO.
11. Для больших обменов не пиши всё в одной транзакции — дели на пакеты.
12. Автоматические обмены по расписанию делай через регламентные задания и проектируй их безопасными к повторному запуску.

## 11. БСП и прикладные подсистемы

1. Для печати используй отдельный общий модуль с суффиксом `Печать`.
2. Для файлов используй БСП `ПрисоединённыеФайлы`.
3. Для почты используй БСП `Почта`.
4. Для миграций используй `ОбновлениеИнформационнойБазы`.

## 12. База и бэкапы

1. Для рабочей файловой базы основной бэкап это копия `1Cv8.1CD` или всего каталога базы.
2. `dt` используй в первую очередь для переноса, обмена и учебных снимков, а не как единственный рабочий бэкап.

## 13. Самопроверка перед выдачей кода

1. Пройди все применимые чек-листы из `Part D` файла `RULES_1C.md`.
2. Каждый пункт должен быть отмечен как выполненный.
3. Если есть хотя бы одно нарушение, итоговый код выдавать нельзя.

## 14. Когда открывать полный стандарт

Сразу переходи к `RULES_1C.md`, если задача затрагивает:
1. сложную СКД;
2. редкие объекты метаданных;
3. расширения конфигурации;
4. CI/CD, тестирование, Vanessa Automation;
5. наблюдаемость, технологический журнал, OpenMetrics;
6. интеграции повышенной сложности;
7. спорный или нетиповой случай, где этого файла уже недостаточно.

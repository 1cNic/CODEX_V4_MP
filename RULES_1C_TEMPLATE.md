# Промышленный стандарт разработки 1С

> **Версия:** 4.0-draft | **Статус:** Каркас
>
> Этот документ задает полный стандарт разработки на `1С:Предприятие 8.3` для задач любой сложности.
> Он работает совместно с `RULES_1C_QUICK.md`, который является коротким обязательным ядром (`CORE`).

---

## 0. Мандат документа

### 0.1 Как использовать этот документ

- `RULES_1C_QUICK.md` — короткий обязательный минимум для ежедневной работы.
- `RULES_1C.md` — полный стандарт с развилкой по профилям задач.
- `CODEX_MISTAKES_1C.md` — локальная память проекта по ошибкам и антипримерам.
- `PATTERNS_1C.md` — локальные удачные решения и повторно используемые эталоны.

### 0.2 Профили применимости

Каждое правило должно иметь:
- профиль применимости;
- уровень строгости.

Профили:
- `ALL` — правило действует всегда;
- `EDU` — учебные и тестовые задачи;
- `PROD` — прикладная промышленная разработка;
- `ENT` — сложные промышленные сценарии, интеграции, highload, наблюдаемость, CI/CD.

Уровни строгости:
- `MUST` — обязательно;
- `SHOULD` — рекомендуется;
- `MUST NOT` — запрещено.

Пример записи:
- `[ALL][MUST] Не использовать ВЫБРАТЬ *`.
- `[EDU][SHOULD] Для простого учебного отчета допустим запрос + форма + ТаблицаЗначений`.
- `[PROD][MUST] Для пользовательского аналитического отчета с вариантами использовать СКД`.
- `[ENT][SHOULD] Для длительных операций использовать фоновые задания и мониторинг выполнения`.

### 0.3 Как выбирать профиль задачи

| Тип задачи | Профиль |
|------------|---------|
| Учебная обработка, учебный документ, тестовое задание | `ALL + EDU` |
| Доработка типовой конфигурации, прикладная разработка в боевой базе | `ALL + PROD` |
| Интеграции, обмены, сложная СКД, фоновые задания, CI/CD, мониторинг | `ALL + PROD + ENT` |

## 1. Протокол работы AI по 1С

- `[ALL][MUST]` Перед любыми изменениями по 1С сначала сверяться с `CODEX_MISTAKES_1C.md`.
- `[ALL][MUST]` Потом проверять `PATTERNS_1C.md`, если в проекте уже есть подходящий успешный пример.
- `[ALL][MUST]` Потом искать локальный эталон в текущем проекте, `samples_1c` или `tools/reference_indexes`.
- `[ALL][MUST]` После выбора подхода делать одну реализацию без архитектурных прыжков.
- `[ALL][MUST]` Если точного эталона нет, честно остановиться и сказать, что нужен образец, а не сочинять.

### 1.1 Что считать локальным эталоном

- рабочий файл из текущего проекта;
- подтвержденный шаблон из `samples_1c`;
- индекс или reference-файл проекта;
- только после этого — внешние источники.

### 1.2 Что запрещено угадывать

- `[ALL][MUST NOT]` XML-структуру метаданных;
- `[ALL][MUST NOT]` управляемые формы и события форм;
- `[ALL][MUST NOT]` имена форм и параметры `ОткрытьФорму(...)`;
- `[ALL][MUST NOT]` `cfg:CatalogRef...`, `cfg:CatalogObject...`, `cfg:Document...`, `cfg:EnumRef...`;
- `[ALL][MUST NOT]` `WindowOpeningMode`, `Attributes`, `Commands`, `ChildItems`, `StandardAttributes`;
- `[ALL][MUST NOT]` подменять стандартное поведение формы пользовательским кодом без подтвержденного локального примера.

# PART A — УНИВЕРСАЛЬНОЕ ЯДРО (`ALL`)

## 2. Запросы

- `[ALL][MUST]` Не использовать `ВЫБРАТЬ *`.
- `[ALL][MUST]` Все поля перечислять явно и задавать алиасы через `КАК`.
- `[ALL][MUST]` Все значения передавать только через параметры `&ИмяПараметра`.
- `[ALL][MUST]` Устанавливать параметры через `Запрос.УстановитьПараметр()`.
- `[ALL][MUST]` Отборы размещать как можно ближе к источнику данных.
- `[ALL][MUST NOT]` Применять функции к полям в `ГДЕ`, если можно отобрать по диапазону или индексируемому значению.
- `[ALL][SHOULD]` Не сортировать в запросе без реальной необходимости.
- `[ALL][MUST]` Перед обработкой результата проверять, что выборка не пустая.
- `[ALL][MUST]` После `ЛЕВОЕ СОЕДИНЕНИЕ` не фильтровать поля правой таблицы в `ГДЕ`, если нужно сохранить все строки левого источника.
- `[ALL][MUST]` Для фильтрации по агрегатам использовать `ИМЕЮЩИЕ`, а не `ГДЕ`.
- `[ALL][MUST]` В `СГРУППИРОВАТЬ ПО` указывать реальные имена полей, а не псевдонимы.
- `[ALL][MUST]` Различать `NULL`, пустую строку, пустую ссылку и `Неопределено`; `NULL` проверять через `ЕСТЬ NULL`.
- `[ALL][MUST]` `ПРЕДСТАВЛЕНИЕ()` использовать только для вывода пользователю.
- `[ALL][SHOULD]` Для сложных и повторно используемых выборок предпочитать временные таблицы `ПОМЕСТИТЬ`.

## 3. Код BSL

- `[ALL][MUST]` Код должен быть готов к вставке, без заглушек и комментариев вида «тут ваш код».
- `[ALL][SHOULD]` Повторно используемую логику выносить в общие модули.
- `[ALL][MUST NOT]` Создавать большие универсальные `god`-модули.
- `[ALL][MUST]` Экспортировать только действительно нужные процедуры и функции.
- `[ALL][SHOULD]` Начинать работу со `Ссылки`; к `Объекту` переходить только когда действительно нужно менять данные.
- `[ALL][MUST]` В модуле объекта использовать `ЭтотОбъект`, в модуле формы `Объект`.
- `[ALL][SHOULD]` Документировать экспортные процедуры и функции.
- `[ALL][MUST]` Если поведение кода непонятно, сначала воспроизводить его, а не додумывать.

## 4. Формы, клиент и сервер

- `[ALL][MUST]` UI и обработчики формы держать на клиенте.
- `[ALL][MUST]` Запросы, проведение, расчеты и работу с данными выносить на сервер.
- `[ALL][MUST]` Явно указывать директивы `&НаКлиенте`, `&НаСервере`, `&НаСервереБезКонтекста`.
- `[ALL][SHOULD]` Если контекст формы не нужен, предпочитать `&НаСервереБезКонтекста`.
- `[ALL][SHOULD]` Минимизировать передачу данных между клиентом и сервером.
- `[ALL][MUST NOT]` Размещать бизнес-логику в `ПриОткрытии`, если по сценарию нужна кнопка.
- `[ALL][SHOULD]` Если нужна гарантированно видимая кнопка, добавлять ее явно по локальному эталону.

## 5. Транзакции и производительность

- `[ALL][MUST]` Транзакцию открывать как можно позже и закрывать как можно раньше.
- `[ALL][MUST NOT]` Вызывать интерактивные методы внутри транзакции.
- `[ALL][MUST]` Внутри транзакций использовать `Попытка...Исключение`.
- `[ALL][SHOULD]` Для больших выборок и сервисов использовать пагинацию.
- `[ALL][SHOULD]` При кэшировании продумывать инвалидирование кэша.

## 6. Объекты метаданных и XML

- `[ALL][MUST]` Все правки `Configuration.xml`, объектов метаданных, форм и `Ext/Form.xml` делать только по локальному эталону.
- `[ALL][MUST]` Перед правкой проверять имя объекта, путь файла, `ChildObjects`, формы по умолчанию и ссылки в `Configuration.xml`.
- `[ALL][MUST NOT]` Считать изменившийся текст ошибки доказательством правильного решения.
- `[ALL][SHOULD]` Проверять, не является ли новая ошибка следствием предыдущей импровизации.

## 7. Безопасность и интеграции

- `[ALL][MUST NOT]` Хранить пароли, сертификаты и секреты в открытом виде.
- `[ALL][MUST]` Использовать `ЗащищённоеХранилище` и безопасные механизмы БСП.
- `[ALL][MUST]` Для внешних интеграций использовать HTTPS.
- `[ALL][MUST]` В HTTP-сервисах возвращать корректные коды ответа.
- `[ALL][MUST]` В HTTP-ответах явно задавать `Content-Type` и кодировку.
- `[ALL][MUST]` Тело HTTP-ответа разбирать только после проверки `КодСостояния`.

# PART B — УЧЕБНЫЕ ЗАДАЧИ (`EDU`)

## 8. Принципы учебной разработки

- `[EDU][MUST]` Решение должно в первую очередь соответствовать ТЗ, а не абстрактной “идеальной архитектуре”.
- `[EDU][SHOULD]` Для учебной конфигурации выбирать простое и прозрачное решение.
- `[EDU][MUST NOT]` Тащить тяжелую промышленную архитектуру без прямой необходимости.
- `[EDU][SHOULD]` Предпочитать читаемость, повторяемость и простую отладку.

## 9. Документы, регистры, проведение

- `[EDU][SHOULD]` Для учебного документа держать состав реквизитов и табличных частей минимально достаточным.
- `[EDU][SHOULD]` Для контроля остатков предпочитать простой и прозрачный алгоритм.
- `[EDU][MUST]` Понятные ошибки проведения показывать пользователю явным текстом.
- `[EDU][SHOULD]` Если внутренние имена можно делать русскими без нарушения правил платформы, это допустимо.

## 10. Учебные отчеты

- `[EDU][SHOULD]` Для простого учебного отчета допустим запрос + форма + `ТаблицаЗначений`, если это соответствует ТЗ.
- `[EDU][MUST NOT]` Собирать программную СКД с нуля без подтвержденного локального шаблона.
- `[EDU][SHOULD]` Для отчета по регистру остатков использовать самый прозрачный путь вывода данных.

## 11. Минимально достаточная проверка

- `[EDU][MUST]` Проверять основной пользовательский сценарий.
- `[EDU][SHOULD]` Проверять хотя бы один негативный сценарий.
- `[EDU][SHOULD]` Фиксировать, что было проверено вручную пользователем.

# PART C — ПРИКЛАДНАЯ ПРОМЫШЛЕННАЯ РАЗРАБОТКА (`PROD`)

## 12. Общие принципы

- `[PROD][MUST]` Проектировать решение с учетом обновляемости и сопровождения.
- `[PROD][MUST]` Использовать БСП и типовые механизмы, если они уже являются частью прикладного решения.
- `[PROD][MUST NOT]` Создавать дублирующие механизмы там, где уже есть штатный механизм платформы или БСП.

## 13. БСП, типовые конфигурации, расширения

- `[PROD][MUST]` Соблюдать суффиксы модулей БСП и их назначение.
- `[PROD][MUST]` Для доработок типовых конфигураций предпочитать расширения, если это допустимо по задаче.
- `[PROD][MUST]` Не ломать обновляемость ради локального удобства.
- `[PROD][SHOULD]` Использовать подписки и точки расширения вместо прямого вмешательства в типовой код.

## 14. Отчеты и пользовательская аналитика

- `[PROD][MUST]` Для пользовательских аналитических отчетов с настройками и вариантами использовать СКД.
- `[PROD][MUST]` Если меняются только поля, оформление, сортировка, отборы или представление, делать новый вариант отчета, а не новый отчет.
- `[PROD][SHOULD]` Проектировать отчет так, чтобы пользовательские настройки и варианты были устойчивыми к доработкам.

## 15. Ошибки, логирование, сопровождение

- `[PROD][MUST]` Разделять сообщение пользователю и техническую ошибку для разработчика/администратора.
- `[PROD][MUST]` Ключевые ошибки фиксировать в журнале регистрации.
- `[PROD][SHOULD]` Важные прикладные операции делать наблюдаемыми и диагностируемыми.

## 16. Тестирование и релизная дисциплина

- `[PROD][MUST]` Перед релизом проверять ключевые сценарии.
- `[PROD][SHOULD]` Автоматизировать синтаксический контроль и тесты.
- `[PROD][SHOULD]` Проектировать миграции данных безопасными к повторному запуску.

# PART D — СЛОЖНЫЕ ПРОМЫШЛЕННЫЕ СЦЕНАРИИ (`ENT`)

## 17. Интеграции

- `[ENT][MUST]` Для сложных интеграций явно проектировать формат обмена, обработку ошибок, идемпотентность и повторный запуск.
- `[ENT][SHOULD]` При серии вызовов одного сервиса переиспользовать `HTTPСоединение`.
- `[ENT][SHOULD]` Для больших JSON использовать потоковые `ЧтениеJSON` и `ЗаписьJSON`.

## 18. Асинхронность, фоновые задания, длительные операции

- `[ENT][SHOULD]` Все операции дольше 5 секунд выносить в фоновые задания или асинхронный сценарий.
- `[ENT][MUST]` Проектировать безопасную повторную постановку и завершение фоновых заданий.
- `[ENT][SHOULD]` Обеспечивать контроль статуса, отмены и диагностики выполнения.

## 19. Кэширование, блокировки, нагрузка

- `[ENT][SHOULD]` Для кэша проектировать TTL и инвалидирование.
- `[ENT][MUST]` При конкурентной работе учитывать конфликты блокировок и повторные попытки.
- `[ENT][SHOULD]` Для высоконагруженных выборок использовать пагинацию, индексацию и предварительные наборы данных.

## 20. CI/CD и наблюдаемость

- `[ENT][SHOULD]` Автоматизировать синтаксический контроль, тесты и сборку.
- `[ENT][SHOULD]` Настраивать технологический журнал и ключевые точки логирования.
- `[ENT][SHOULD]` Использовать метрики и мониторинг для длительных и критичных процессов.

# PART E — ВЫБОР АРХИТЕКТУРЫ ОТЧЕТА

## 21. Когда делать простой отчет на форме

- `[EDU][SHOULD]` Если задача учебная, отчет простой, пользователю не нужны варианты и расширенные настройки.
- `[ALL][SHOULD]` Если нужен быстрый и прозрачный вывод данных без сложной аналитики.

## 22. Когда делать СКД

- `[PROD][MUST]` Если нужны варианты отчета, пользовательские настройки, группировки, ресурсы, расшифровки, отборы и типовое поведение.
- `[PROD][SHOULD]` Если отчет будет развиваться и обслуживаться в прикладной системе.

## 23. Когда нужен ТабличныйДокумент

- `[ALL][SHOULD]` Если нужен печатный или близкий к печатной форме результат.
- `[PROD][SHOULD]` Если отчет должен соответствовать утвержденному макету.

# PART F — XML И УПРАВЛЯЕМЫЕ ФОРМЫ

## 24. Жесткие правила

- `[ALL][MUST]` Перед изменением XML и форм сначала находить точный локальный эталон.
- `[ALL][MUST]` Проверять `Configuration.xml`, путь файла, имя объекта, `ChildObjects`, формы по умолчанию и связанные XML-файлы.
- `[ALL][MUST NOT]` Угадывать имя формы открытия в `ОткрытьФорму(...)`.
- `[ALL][MUST NOT]` Угадывать имя события формы в XML.
- `[ALL][MUST NOT]` Угадывать `WindowOpeningMode`, `Attributes`, `Commands`, `ChildItems`, `StandardAttributes`.
- `[ALL][MUST NOT]` Угадывать QName и `cfg:...` ссылки.

## 25. Проверка перед передачей проекта на загрузку

- `[ALL][MUST]` Новый объект добавлен в `Configuration.xml`, если это новый объект.
- `[ALL][MUST]` Имя объекта совпадает с путем файла.
- `[ALL][MUST]` Не осталось хвостов от старого имени после переименования.
- `[ALL][MUST]` Нет автозапуска логики из формы, если по сценарию нужна кнопка.
- `[ALL][MUST]` Нет обращения к несуществующим функциям платформы.

# PART G — SELF-CHECK

## 26. Общий принцип

- `[ALL][MUST]` Перед выдачей итогового кода или правок проходить все применимые чек-листы.
- `[ALL][MUST]` Не выдавать итоговый код, если хотя бы один критичный пункт чек-листа не выполнен.

## 27. CORE checklist (`ALL`)

Пример минимального ядра:
- нет `ВЫБРАТЬ *`;
- все параметры передаются через `УстановитьПараметр`;
- клиент и сервер разделены;
- XML и формы правятся только по эталону;
- код готов к вставке;
- ошибка пользователя отделена от технической причины.

## 28. EDU checklist

Пример:
- решение соответствует ТЗ;
- архитектура не переусложнена;
- сценарий прост и прозрачен;
- есть понятный пользовательский результат.

## 29. PROD checklist

Пример:
- учтены БСП и типовые механизмы;
- решение сопровождаемо и обновляемо;
- учтены логирование и ошибки;
- отчет/интеграция соответствуют прикладной задаче.

## 30. ENT checklist

Пример:
- учтены фоновые задания и длительные операции;
- интеграция устойчива к повторному запуску;
- есть наблюдаемость, мониторинг или диагностические точки;
- учтены производительность и конкурентная работа.

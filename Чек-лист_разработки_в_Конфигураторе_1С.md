# Чек-лист реализации в Конфигураторе 1С

Ниже приведен практический порядок, в котором удобнее собирать решение по тестовому заданию. Он рассчитан на быстрый выход к рабочему результату без лишнего усложнения.

## 1. Создать основу конфигурации

1. Создать новую информационную базу.
2. Открыть конфигурацию в Конфигураторе.
3. Убедиться, что используется управляемое приложение.
4. Сразу задать понятные имена объектов, чтобы потом не путаться.

## 2. Создать перечисление

Создать перечисление `ВидыОперацийТоваров` со значениями:

1. `Приход`
2. `Расход`

Оно понадобится документу для одного универсального механизма движения.

## 3. Создать справочник `Номенклатура`

Создать справочник `Номенклатура`.

Реквизиты:

1. `Артикул`
2. `НаименованиеПолное` или оставить только стандартное `Наименование`, если не хочется усложнять

Практично сделать так:

- стандартный реквизит `Наименование` использовать как основное название товара;
- отдельный реквизит `Артикул` хранить как строку.

Что настроить:

1. Форму списка.
2. Форму элемента.
3. Командный интерфейс, чтобы справочник был доступен из панели разделов.

## 4. Создать подчиненный справочник `ХарактеристикиНоменклатуры`

Создать справочник `ХарактеристикиНоменклатуры`.

Настройки:

1. Подчинение справочнику `Номенклатура`.
2. Иерархия не нужна.
3. Автонумерация не нужна.

Реквизиты:

1. `ВидХарактеристики` — строка или перечисление.
2. `ЗначениеХарактеристики` — строка.

Если хочется сделать аккуратнее, можно добавить перечисление `ВидыХарактеристик`:

1. `Цвет`
2. `Размер`

Это будет лучше, чем хранить вид характеристики строкой.

Что важно:

- у элемента справочника владелец будет конкретная `Номенклатура`;
- это и даст нужную связь для отбора в документе.

## 5. Создать регистр накопления `ОстаткиТоваров`

Создать регистр накопления с видом `Остатки`.

Измерения:

1. `Номенклатура` — ссылка на справочник `Номенклатура`
2. `Характеристика` — ссылка на справочник `ХарактеристикиНоменклатуры`

Ресурсы:

1. `Количество` — число

Этого достаточно. Лишние измерения пока не добавлять.

## 6. Создать документ `ДвижениеТоваров`

Создать документ `ДвижениеТоваров`.

Реквизиты шапки:

1. `ВидОперации` — перечисление `ВидыОперацийТоваров`

Табличная часть `Товары`:

1. `Номенклатура`
2. `Характеристика`
3. `Количество`

После этого создать:

1. Основную форму документа.
2. Форму списка документа.

## 7. Настроить форму документа

На форме документа нужно добиться трех вещей:

1. Выбор номенклатуры в строке.
2. Отбор характеристик по выбранной номенклатуре.
3. Вывод артикула отдельной колонкой.

Что сделать на форме:

1. Открыть форму документа.
2. Вывести табличную часть `Товары`.
3. Добавить колонку `Номенклатура`.
4. Добавить колонку `Характеристика`.
5. Добавить колонку `Количество`.
6. Добавить отдельную колонку для артикула.

Для колонки артикула есть два удобных пути:

1. Сделать реквизит формы, связанный с текущей строкой.
2. Для тестового задания проще добавить в ТЧ реквизит `Артикул` и заполнять его автоматически при выборе номенклатуры.

Для быстрого выполнения ТЗ второй путь проще. Да, это дублирование данных, но для тестового задания он удобен и надежен.

Тогда в табличную часть `Товары` добавить еще реквизит:

1. `Артикул` — строка

И на форме просто вывести его отдельной колонкой.

## 8. Настроить поведение строки ТЧ

Нужно реализовать логику:

- выбрали номенклатуру;
- в строке автоматически заполнился артикул;
- характеристика отбирается только по этой номенклатуре;
- если номенклатуру сменили, характеристика очищается.

Что сделать:

1. В модуле формы документа обработать изменение `Номенклатуры` в строке ТЧ.
2. При изменении:
   - записать `Артикул = Номенклатура.Артикул`;
   - очистить `Характеристику`;
   - установить отбор для выбора характеристик по владельцу.

Обычно это делается через обработчик начала выбора или через параметры выбора поля формы.

Логика должна быть такой:

- если в строке выбрана номенклатура, показывать только характеристики, где владелец = эта номенклатура;
- если номенклатура не выбрана, характеристику выбирать нельзя или список должен быть пустым.

## 9. Реализовать проведение документа

У документа открыть:

1. Конструктор движений.
2. Добавить движения по регистру `ОстаткиТоваров`.

Логика проведения:

1. Для каждой строки ТЧ сформировать движение.
2. Если `ВидОперации = Приход`, движение типа `Приход`.
3. Если `ВидОперации = Расход`, движение типа `Расход`.
4. Записать:
   - `Номенклатура`
   - `Характеристика`
   - `Количество`

Это можно сначала сделать конструктором, потом руками поправить код.

## 10. Реализовать контроль остатков

Это самый важный блок после движений.

Что нужно:

1. При `Приход` документ проводится без проверки.
2. При `Расход` нужно проверить остатки по каждой строке.
3. Если остатка не хватает, проведение отменяется.

Удобный порядок:

1. В обработке проведения собрать все строки документа.
2. Получить остатки из регистра на момент документа.
3. Сравнить остаток и количество расхода.
4. Если остаток меньше требуемого:
   - установить `Отказ = Истина`;
   - сообщить понятный текст ошибки.

Текст ошибки лучше делать такого вида:

`Недостаточно остатка по номенклатуре "Футболка", характеристика "Размер L". Доступно: 2, требуется: 5.`

Это сразу показывает, что контроль действительно работает.

## 11. Сделать тестовую логику для наполнения остатка

Чтобы контроль остатков можно было продемонстрировать:

1. Создать документ `Приход`.
2. Провести его.
3. Создать документ `Расход` на меньшее количество.
4. Провести его успешно.
5. Создать документ `Расход` на большее количество.
6. Получить ошибку.

Это стоит прогнать до отчета и HTTP-сервиса.

## 12. Создать отчет на СКД

Создать отчет `ОстаткиТоваровОтчет`.

Дальше:

1. Открыть схему компоновки данных.
2. Добавить набор данных на основе регистра накопления.
3. Лучше всего использовать виртуальную таблицу остатков и оборотов.
4. Вывести поля:
   - `Номенклатура`
   - `Характеристика`
   - `КоличествоНачальныйОстаток`
   - `КоличествоПриход`
   - `КоличествоРасход`
   - `КоличествоКонечныйОстаток`

Если хочется сделать чуть лучше:

1. Добавить `Артикул` через соединение с `Номенклатурой`.
2. Добавить отбор по периоду.

Но для минимального результата это не обязательно.

## 13. Создать HTTP-сервис

Создать HTTP-сервис, например `ОстаткиAPI`.

В нем создать шаблон URL, например:

1. `/stocks`

Пусть запрос будет таким:

`/stocks?articles=TSH-001,TSH-002`

Что должен делать обработчик:

1. Получить строку параметра `articles`.
2. Разбить ее по запятым в массив.
3. Найти номенклатуру по артикулу.
4. Получить остатки по регистру.
5. Сформировать JSON.
6. Вернуть ответ с типом `application/json`.

Состав JSON:

1. `Дата`
2. `Артикул`
3. `Номенклатура`
4. `Характеристика`
5. `Количество`

## 14. Проверка сквозного сценария

Перед выгрузкой обязательно пройти весь путь руками:

1. Создать номенклатуру.
2. Создать подчиненные характеристики.
3. Создать приходный документ.
4. Провести его.
5. Создать расходный документ с допустимым количеством.
6. Провести его.
7. Создать расходный документ с превышением остатка.
8. Убедиться, что появляется ошибка.
9. Построить отчет.
10. Вызвать HTTP-сервис и проверить JSON.

## 15. Финальная упаковка

В конце:

1. Обновить конфигурацию базы данных.
2. Проверить, что все объекты открываются без ошибок.
3. Выгрузить в `.cf` или `.dt`.
4. При передаче работодателю при желании добавить короткий `README` с описанием:
   - как открыть;
   - какие объекты есть;
   - как проверить HTTP-сервис.

## Практический минимум по приоритету

Если времени мало, имеет смысл идти в таком порядке:

1. `Номенклатура`
2. `ХарактеристикиНоменклатуры`
3. `ДвижениеТоваров`
4. `ОстаткиТоваров`
5. Проведение
6. Контроль остатков
7. Отчет СКД
8. HTTP-сервис
9. Выгрузка `.cf`

## Что лучше не усложнять

Для тестового задания не стоит тратить время на:

1. Сложные права доступа.
2. Красивые подсистемы.
3. Продвинутый UI.
4. Универсальные механизмы характеристик.
5. Сложную валидацию всех реквизитов.

Лучше сделать простой, законченный и рабочий контур.

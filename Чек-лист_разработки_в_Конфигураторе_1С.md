# Чек-лист разработки в Конфигураторе 1С

Ниже приведен практический порядок реализации тестового задания по 1С. Чек-лист выровнен под исходное ТЗ и ориентирован на нормальную прикладную архитектуру без лишнего усложнения.

Для каждого пункта этого чек-листа обязательно применять правила из `CODEX_STRICT_1C.md`.

Обозначения статуса:
- `Не выполнено`
- `Частично выполнено`
- `Выполнено`

Важно:
- этот чек-лист описывает целевую структуру решения;
- текущий проект может временно отличаться от целевой структуры;
- если текущее состояние проекта расходится с этим документом, сначала нужно явно зафиксировать расхождение, а потом принимать решение: исправлять проект или корректировать чек-лист.

## 1. Создать основу конфигурации

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

1. Создать новую информационную базу.
2. Открыть конфигурацию в Конфигураторе.
3. Убедиться, что используется управляемое приложение.
4. Сразу задать понятные имена объектов.
5. Сразу вести проект как выгрузку в файлы, если задача делается в файловой структуре конфигурации.

## 2. Создать справочник `Номенклатура`

Статус в текущем проекте: `Выполнено`.
Что уже есть:
- создан справочник `Номенклатура`;
- есть реквизит `Артикул`;
- добавлены форма элемента и форма списка.
Что еще проверить или довести:
- корректность открытия формы элемента и формы списка;
- доступность справочника из интерфейса;
- отсутствие лишних или временных решений в формах.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Реквизиты:
1. `Артикул`
2. стандартное `Наименование`

Что важно по архитектуре:
- `Наименование` использовать как основное имя товара;
- `Артикул` хранить отдельным реквизитом;
- не усложнять объект дополнительными реквизитами без необходимости ТЗ.

Что настроить:
1. Форму списка.
2. Форму элемента.
3. Командный интерфейс или иной стандартный способ открытия.

## 3. Создать подчиненный справочник `Характеристики`

Статус в текущем проекте: `Выполнено`.
Что уже есть:
- создан подчиненный справочник `Характеристики`;
- настроен владелец `Номенклатура`;
- есть форма элемента;
- есть реквизиты `Цвет` и `Размер`.
Что еще проверить или довести:
- корректность стандартного владельца;
- отсутствие лишней универсализации;
- корректность формы элемента.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Целевая модель по ТЗ:
1. Справочник называется `Характеристики`.
2. Справочник подчинен `Номенклатура`.
3. В справочнике есть реквизиты `Цвет` и `Размер`.

Что важно по архитектуре:
- для данного ТЗ не нужно вводить универсальную модель `ВидХарактеристики` / `ЗначениеХарактеристики`;
- не нужно добавлять отдельное перечисление видов характеристик, если задача решается реквизитами `Цвет` и `Размер`;
- владелец характеристики должен использоваться для отбора характеристик по номенклатуре в документе.

## 4. Создать перечисление `ВидыОперацийТоваров`

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Значения:
1. `Приход`
2. `Расход`

Что важно:
- перечисление нужно создать до завершения документа и проведения;
- оно определяет направление движения по регистру;
- если временно документ разрабатывается без этого перечисления, это нужно считать промежуточным состоянием, а не целевой архитектурой.

## 5. Создать документ `ДвижениеТоваров`

Статус в текущем проекте: `Выполнено`.
Что уже есть:
- создан документ `ДвижениеТоваров`;
- есть табличная часть `Товары`;
- в табличной части есть поля `Номенклатура`, `Характеристика`, `Количество`.
 - добавлены форма документа и форма списка.
Что еще отсутствует или требует проверки:
- соответствие документа итоговому сценарию ТЗ.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Реквизиты шапки:
1. `ВидОперации` — перечисление `ВидыОперацийТоваров`

Табличная часть `Товары`:
1. `Номенклатура`
2. `Характеристика`
3. `Количество`

Что важно по архитектуре:
- сначала нужно зафиксировать полный состав документа, потом уже строить проведение;
- не стоит считать документ завершенным, если у него еще нет реквизита, который определяет бизнес-логику проведения;
- формы документа следует создавать после фиксации состава реквизитов.

## 6. Настроить форму документа

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

На форме документа нужно обеспечить:
1. Выбор номенклатуры в строке.
2. Выбор характеристики только в контексте выбранной номенклатуры.
3. Вывод артикула отдельной колонкой на форме табличной части.

Что сделать:
1. Создать основную форму документа.
2. Вывести табличную часть `Товары`.
3. Вывести колонки `Номенклатура`, `Характеристика`, `Количество`.
4. Вывести отдельную колонку `Артикул`.

Что важно по архитектуре:
- основным вариантом считать вывод артикула на форме без дублирования данных в документе;
- добавление реквизита `Артикул` в табличную часть допустимо только как осознанное упрощение, если без этого невозможно быстро завершить тестовое задание;
- если используется упрощение с дублированием, это нужно явно пометить как компромисс.

## 7. Настроить поведение строки табличной части

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Нужно реализовать логику:
- выбрали номенклатуру;
- характеристика выбирается только для этой номенклатуры;
- если номенклатуру сменили, характеристика очищается;
- на форме корректно показывается артикул выбранной номенклатуры.

Что сделать:
1. В модуле формы документа обработать изменение номенклатуры в строке ТЧ.
2. При изменении номенклатуры очистить характеристику.
3. Настроить отбор характеристик по владельцу.
4. Настроить показ артикула в отдельной колонке формы.

Что важно:
- UI-логика должна оставаться в форме;
- работа с данными и движениями должна оставаться на сервере;
- не нужно переносить логику проведения в форму.

## 8. Создать регистр накопления `ОстаткиТоваров`

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Тип регистра:
1. `Остатки`

Измерения:
1. `Номенклатура` — ссылка на справочник `Номенклатура`
2. `Характеристика` — ссылка на справочник `Характеристики`

Ресурсы:
1. `Количество` — число

Что важно по архитектуре:
- регистр должен хранить именно то, по чему нужны остатки и отчеты;
- не нужно добавлять лишние измерения;
- направление движения определяется документом и перечислением `ВидыОперацийТоваров`.

## 9. Реализовать проведение документа

Статус в текущем проекте: `Выполнено`.
Что уже есть:
- документ уже связан с регистром как регистратор;
- в проекте есть первичная логика записи движений.
Что еще довести:
- проверить корректность имен табличной части и полей;
- убрать временные упрощения, если они были внесены.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Логика проведения:
1. Для каждой строки ТЧ сформировать движение по регистру `ОстаткиТоваров`.
2. Если `ВидОперации = Приход`, делать движение `Приход`.
3. Если `ВидОперации = Расход`, делать движение `Расход`.
4. Записывать:
   - `Номенклатура`
   - `Характеристика`
   - `Количество`

Что важно:
- проведение должно быть реализовано в модуле объекта документа;
- промежуточный вариант “все движения как приход” не является целевым решением;
- перед финальной сдачей логика проведения должна соответствовать ТЗ полностью.

Текущее замечание по проекту:
- пункт нельзя считать завершенным по стандартам 1С, пока документ не переведен на штатную прикладную логику `Приход` / `Расход`.

## 10. Реализовать контроль остатков

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Требования:
1. При `Приход` документ проводится без проверки остатков.
2. При `Расход` нужно проверить остатки по каждой строке.
3. Если остатка не хватает, проведение отменяется.

Рекомендуемый порядок:
1. В обработке проведения собрать все строки документа.
2. Получить остатки из регистра на момент документа.
3. Сравнить остаток и количество расхода.
4. Если остаток меньше требуемого:
   - установить `Отказ = Истина`;
   - вывести понятный текст ошибки.

Что важно по архитектуре:
- контроль остатков является частью логики проведения, а не отдельной клиентской проверкой;
- сообщение об ошибке должно быть прикладным и понятным пользователю;
- проверка должна учитывать номенклатуру и характеристику вместе.

## 11. Протестировать сценарий прихода и расхода

Статус в текущем проекте: `Выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Нужно проверить руками:
1. Создать документ `Приход`.
2. Провести его.
3. Убедиться, что движения записались в регистр.
4. Создать документ `Расход` на меньшее количество.
5. Провести его успешно.
6. Создать документ `Расход` на большее количество.
7. Получить ошибку контроля остатков.

Что важно:
- этот шаг должен выполняться до отчета и HTTP-сервиса;
- без него дальше легко строить решения поверх неподтвержденной базы.

## 13. Создать HTTP-сервис

Статус в текущем проекте: `Не выполнено`.
Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Назначение:
1. вернуть остатки по списку артикулов

Что должен делать сервис:
1. Получить список артикулов из параметров запроса.
2. Найти номенклатуру по артикулу.
3. Получить остатки по регистру.
4. Сформировать JSON.
5. Вернуть ответ с корректным `Content-Type`.

Состав ответа:
1. `Дата`
2. `Артикул`
3. `Номенклатура`
4. `Характеристика`
5. `Количество`

Что важно по архитектуре:
- формат ответа должен быть `application/json`;
- обработчик должен возвращать только актуальные остатки;
- структура ответа должна быть стабильной и понятной.

## Что лучше не усложнять

Правило выполнения: строго по `CODEX_STRICT_1C.md`.

Для этого тестового задания не стоит тратить время на:
1. сложные права доступа;
2. лишние подсистемы;
3. продвинутый UI;
4. универсальные механизмы характеристик;
5. лишние реквизиты и абстракции, которых нет в ТЗ;
6. избыточную валидацию всех реквизитов.

Лучше сделать простой, законченный и рабочий прикладной контур.

# Паттерны 1С

Этот файл хранит успешные решения проекта как локальные эталоны.

Использование:
- сначала читать `CODEX_MISTAKES_1C.md`;
- потом искать подходящий паттерн в этом файле;
- только после этого повторять решение в новом месте проекта.

## 1. Подчиненный справочник характеристик с автосборкой наименования

- Где используется:
  - [Catalogs/Характеристики.xml](C:/Users/user/CODEX_V4_MP/Catalogs/Характеристики.xml)
  - [ObjectModule.bsl](C:/Users/user/CODEX_V4_MP/Catalogs/Характеристики/Ext/ObjectModule.bsl)
- Задача:
  - хранить характеристики номенклатуры в подчиненном справочнике;
  - автоматически собирать `Наименование` из владельца и реквизитов характеристики.
- Правильный подход:
  - использовать подчиненный справочник с `Owner`;
  - обновлять `Наименование` в `ПередЗаписью`;
  - собирать строку из заполненных частей через `СтрСоединить`.
- Почему это эталон:
  - решение простое;
  - не требует лишних формул в запросах;
  - соответствует учебной архитектуре текущего ТЗ.
- Что проверять перед повторным использованием:
  - у объекта действительно есть стандартный реквизит `Owner`;
  - собираемые реквизиты существуют в метаданных;
  - имя объекта и модуль совпадают с путями проекта.
- Чего не делать:
  - не собирать `Наименование` на форме;
  - не дублировать ту же логику в нескольких модулях.

## 2. Документ движения товаров с русскими внутренними именами

- Где используется:
  - [Documents/ДвижениеТоваров.xml](C:/Users/user/CODEX_V4_MP/Documents/ДвижениеТоваров.xml)
- Задача:
  - хранить документ прихода и расхода с табличной частью товаров.
- Правильный подход:
  - использовать русские внутренние имена, если это не противоречит правилам 1С;
  - держать состав документа простым:
    - реквизит `ВидОперации`;
    - табличная часть `Товары`;
    - колонки `Номенклатура`, `Характеристика`, `Количество`.
- Почему это эталон:
  - имена совпадают с предметной областью;
  - код BSL становится читаемее;
  - меньше риска путаницы между внутренними именами и синонимами.
- Что проверять перед повторным использованием:
  - после переименования обновлены все `DataPath`, модули, формы и ссылки в XML;
  - не осталось старых латинских имен в проекте.
- Чего не делать:
  - не переименовывать только часть структуры;
  - не оставлять старые имена в формах и запросах.

## 3. Проведение документа через один явный реквизит вида операции

- Где используется:
  - [Documents/ДвижениеТоваров.xml](C:/Users/user/CODEX_V4_MP/Documents/ДвижениеТоваров.xml)
  - [ObjectModule.bsl](C:/Users/user/CODEX_V4_MP/Documents/ДвижениеТоваров/Ext/ObjectModule.bsl)
  - [Enums/ВидыОперацийТоваров.xml](C:/Users/user/CODEX_V4_MP/Enums/ВидыОперацийТоваров.xml)
- Задача:
  - проводить один документ как `Приход` или `Расход`.
- Правильный подход:
  - вынести вид операции в отдельное перечисление;
  - в документе хранить `ВидОперации`;
  - в модуле объекта получать `ВидДвиженияНакопления` через отдельную функцию.
- Почему это эталон:
  - логика проведения читается явно;
  - не нужно строить два разных документа для учебного сценария;
  - проще тестировать.
- Что проверять перед повторным использованием:
  - перечисление зарегистрировано в `Configuration.xml`;
  - реквизит документа ссылается на правильный тип `cfg:EnumRef...`;
  - документ назначен регистратором нужного регистра.
- Чего не делать:
  - не кодировать приход/расход строками;
  - не размазывать выбор вида движения по нескольким местам.

## 4. Контроль остатков в учебной конфигурации

- Где используется:
  - [ObjectModule.bsl](C:/Users/user/CODEX_V4_MP/Documents/ДвижениеТоваров/Ext/ObjectModule.bsl)
- Задача:
  - не допускать расхода сверх доступного остатка.
- Правильный подход:
  - агрегировать строки документа в `ТаблицаЗначений`;
  - получать остаток отдельным параметризованным запросом по регистру;
  - исключать текущий документ через `Регистратор <> &Ссылка`;
  - при нехватке формировать понятный текст ошибки.
- Почему это эталон:
  - решение прозрачное;
  - его легко читать и отлаживать;
  - оно подходит именно для учебной базы без лишней универсальности.
- Что проверять перед повторным использованием:
  - запрос использует параметры через `УстановитьПараметр`;
  - контроль срабатывает только для расхода;
  - движения текущего документа не искажают проверку при перепроведении.
- Чего не делать:
  - не передавать `ТаблицаЗначений` в запрос ради этой задачи;
  - не усложнять контроль виртуальными таблицами без локального эталона.

## 5. Понятный текст отказа проведения

- Где используется:
  - [ObjectModule.bsl](C:/Users/user/CODEX_V4_MP/Documents/ДвижениеТоваров/Ext/ObjectModule.bsl)
- Задача:
  - показывать пользователю в системном окне понятную причину отказа проведения.
- Правильный подход:
  - собирать текст ошибки заранее;
  - при отказе делать `ВызватьИсключение ТекстОшибки`;
  - `Сообщить()` использовать только как дополнительный канал, если нужно.
- Почему это эталон:
  - пользователь видит причину прямо в основном окне ошибки;
  - это лучше обычного `[ОшибкаХранимыхДанных]`.
- Что проверять перед повторным использованием:
  - текст ошибки формируется до `ВызватьИсключение`;
  - при нескольких ошибках строки можно объединять через `Символы.ПС`.
- Чего не делать:
  - не ограничиваться только `Сообщить()`;
  - не оставлять общий текст без причины.

## 6. Автоподстановка характеристики в форме документа

- Где используется:
  - [Module.bsl](C:/Users/user/CODEX_V4_MP/Documents/ДвижениеТоваров/Forms/ФормаДокумента/Ext/Form/Module.bsl)
- Задача:
  - при выборе номенклатуры автоматически подставлять первую доступную характеристику.
- Правильный подход:
  - на клиенте реагировать только на изменение строки;
  - на сервере подбирать характеристику запросом;
  - пустую ссылку тоже получать на сервере.
- Почему это эталон:
  - корректно разделены клиент и сервер;
  - нет обращения к менеджерам метаданных из клиентского кода формы;
  - логика короткая и предсказуемая.
- Что проверять перед повторным использованием:
  - имя обработчика совпадает с именем элемента формы;
  - сортировка характеристики задана явно;
  - при пустой номенклатуре возвращается пустая ссылка.
- Чего не делать:
  - не вызывать `Справочники.*` из `&НаКлиенте`, если это не подтверждено локальным примером;
  - не подбирать значение прямо в клиентском обработчике запросом.

## 7. Отчет на форме с результатом запроса

- Где используется:
  - [Module.bsl](C:/Users/user/CODEX_V4_MP/Reports/ОстаткиТоваровОтчет/Forms/ФормаОтчета/Ext/Form/Module.bsl)
- Задача:
  - показать остатки и обороты за период в учебном проекте.
- Правильный подход:
  - хранить параметры периода в реквизитах формы;
  - получать данные простым запросом к `ОстаткиИОбороты`;
  - выводить результат в таблицу формы через `ТаблицаЗначений`;
  - сортировать уже после `Выгрузить()`.
- Почему это эталон:
  - решение устойчивее самодельной программной СКД без макета;
  - пользователю проще работать с таблицей результата на форме;
  - архитектура соответствует учебному масштабу проекта.
- Что проверять перед повторным использованием:
  - реквизит формы под `ТаблицаЗначений` инициализирован на сервере;
  - параметры периода передаются в запрос через `УстановитьПараметр`;
  - `ПРЕДСТАВЛЕНИЕ(...)` используется только для вывода.
- Чего не делать:
  - не собирать СКД с нуля без подтвержденного шаблона;
  - не использовать `УПОРЯДОЧИТЬ` по полям `ПРЕДСТАВЛЕНИЕ(...)`.

- Если найден удачный и подтвержденный кусок решения, добавляй его сюда как новый паттерн.
- Если решение оказалось ошибочным или нестабильным, оно не должно попадать в этот файл.
- Если паттерн устарел после переделки архитектуры, обновляй его вместе с рабочими файлами проекта.

# CODEX STRICT 1C

Жесткий протокол работы для задач по `1С:Предприятие 8.3` в этом проекте.

Этот файл не заменяет `RULES_1C.md` и `RULES_1C_QUICK.md`, а задает обязательное поведение при работе в проекте.
Карта ролей документов и быстрый маршрут чтения вынесены в `1C_DOCS_MAP.md`.

## 1. Обязательный порядок чтения перед задачей

Перед любыми изменениями по 1С сначала прочитай:
1. `1C_DOCS_MAP.md`
2. `CODEX_STRICT_1C.md`
3. `CODEX.md`
4. `CODEX_MISTAKES_1C.md`
5. `PATTERNS_1C.md`
6. `RULES_1C_QUICK.md`

Если задача нетиповая, затрагивает формы, XML, метаданные или загрузку конфигурации, дополнительно сверяйся с `RULES_1C.md`.
Новые реальные ошибки и выводы по ним обязательно записывай в `CODEX_MISTAKES_1C.md`, чтобы не повторять их в следующих правках.

Обязательный процесс для всех следующих задач по 1С:
1. сначала `CODEX_MISTAKES_1C.md`;
2. потом `PATTERNS_1C.md`, если в проекте уже есть подходящий успешный пример;
3. потом локальный эталон;
4. потом одна реализация без архитектурных прыжков;
5. если эталона нет, честно говорить, что нужен образец, а не сочинять.

## 2. Что запрещено

Все жесткие запреты по XML, формам, `cfg:...` и догадкам считаются обязательными как в `RULES_1C_QUICK.md`.
В этом файле дополнительно запрещено:
- править "на удачу", если корневая причина не проверена;
- перескакивать к новой реализации, не сверившись с `CODEX_MISTAKES_1C.md` и `PATTERNS_1C.md`;
- заменять локальный эталон предположением только потому, что структура "похожа".

## 3. Как действовать при изменениях в XML и формах

Если меняются:
- `Configuration.xml`
- объект метаданных
- формы
- `Ext/Form.xml`
- структура каталогов объектов
- строки `ОткрытьФорму(...)`

тогда обязательно:
1. найти живой эталон в текущем проекте, `samples_1c` или `tools/reference_indexes`;
2. сверить имена объекта, формы и пути к файлам;
3. сверить ссылки в `Configuration.xml` и `ChildObjects`;
4. только после этого вносить изменения.

Если точный пример не найден, нужно остановиться и явно зафиксировать риск, а не импровизировать.

## 4. Как действовать при ошибках

Если ошибка связана с:
- формой;
- XML;
- открытием формы;
- динамическим списком;
- метаданными;
- именем объекта;
- загрузкой конфигурации;

нужно:
1. сначала проверить корневую причину;
2. потом свериться с типовым примером;
3. только потом править.

Изменение текста ошибки не считается решением, если новая ошибка могла появиться из-за предыдущей импровизации.

## 5. Мини-чек перед выдачей изменений

Перед завершением задачи по 1С проверь:
- имя объекта совпадает с путем файлов;
- объект добавлен в `Configuration.xml`, если это новый объект;
- формы перечислены в `ChildObjects`;
- `DefaultObjectForm` / `DefaultListForm` / `DefaultChoiceForm` указывают на реально существующие формы;
- строки `ОткрытьФорму(...)` совпадают с метаданными один в один;
- в XML нет элементов, добавленных без эталона;
- в форме нет автозапуска логики, если по сценарию нужна кнопка;
- не повторена ошибка из `CODEX_MISTAKES_1C.md`.

Если в ходе задачи была выявлена новая ошибка 1С:
- она должна быть кратко зафиксирована в `CODEX_MISTAKES_1C.md`;
- перед следующей правкой нужно свериться с этим файлом повторно.

## 6. Приоритет поведения

При конфликте ориентируйся так:
1. реальные файлы проекта;
2. `CODEX_MISTAKES_1C.md`;
3. `PATTERNS_1C.md`;
4. `CODEX_STRICT_1C.md`;
5. `CODEX.md`;
6. `RULES_1C_QUICK.md`;
7. `RULES_1C.md`;
7. внешние источники и типовые примеры.

## 7. Короткая формула для запросов

Если пользователь хочет строгого режима, достаточно написать:

`Работай строго по CODEX_STRICT_1C.md: сначала CODEX_MISTAKES_1C.md, потом PATTERNS_1C.md, потом локальный эталон, потом одна реализация без архитектурных прыжков; если эталона нет, честно остановись и скажи, что нужен образец.`
